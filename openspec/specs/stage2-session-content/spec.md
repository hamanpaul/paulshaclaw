## Requirements
### Requirement: Session content extraction from transcripts

Importer adapter SHALL dereference each session's captured `transcript_path` (or agent-equivalent history source) and extract real session content — `user_prompts` and `touched_files` — into the inbox session record, for all three agents (claude-code, codex, copilot-cli). Metadata-only capture (empty `(none)` content) is no longer acceptable.

#### Scenario: Claude transcript extraction
- **WHEN** a claude-code session whose payload carries a `transcript_path` to an existing `.jsonl` is imported
- **THEN** the inbox session SHALL contain the user prompts (from `type=user` lines) and touched files (from `tool_use` Write/Edit `file_path`), non-empty when present in the transcript

#### Scenario: Codex content extraction
- **WHEN** a codex session is imported
- **THEN** the assistant content SHALL be taken from the payload `last_assistant_message`, and user prompts from the rollout transcript when its file exists

#### Scenario: Copilot content extraction
- **WHEN** a copilot-cli session is imported
- **THEN** the importer SHALL locate `~/.copilot/history-session-state/session_<id>_*.json` and extract prompts and assistant content from its `chatMessages`

#### Scenario: Missing transcript degrades gracefully
- **WHEN** the referenced transcript/history file does not exist (e.g. a dead pointer)
- **THEN** the affected content fields SHALL be left empty and the import SHALL complete without error

### Requirement: Per-session title generation

The importer SHALL generate exactly one title per session, at import time, of at most 20 Traditional-Chinese characters, and store it as the session's `assistant_summary` and frontmatter `title`. Title generation MUST NOT block or fail the import when the LLM backend is unavailable.

#### Scenario: Title via local gemma4
- **WHEN** the local gemma4 endpoint is reachable during import
- **THEN** the title SHALL be generated by gemma4, be at most 20 characters, and be recorded with `title_source: gemma4`

#### Scenario: Offline fallback title
- **WHEN** the gemma4 backend is offline or times out
- **THEN** the title SHALL fall back to the first user prompt truncated to 20 characters and be recorded with `title_source: fallback`, and the import SHALL still succeed

### Requirement: Atomizer robustness to project identifier format

The atomizer SHALL distill sessions regardless of the `project` identifier format and MUST NOT silently skip a session because its project contains path separators. URL-form projects (e.g. `github.com/owner/repo`) SHALL be mapped to a path-safe component for filesystem use while the original value is preserved in metadata.

#### Scenario: URL-form project is distilled, not skipped
- **WHEN** an inbox session declares `project: github.com/hamanpaul/serialwrap`
- **THEN** the atomizer SHALL sanitize the project to a path-safe component and produce knowledge slice(s) for that session instead of skipping it

#### Scenario: Original project preserved
- **WHEN** a session with a URL-form project is atomized
- **THEN** the rich original `project` value SHALL remain available in the slice metadata

### Requirement: Backfill of existing sessions

A backfill entrypoint SHALL re-extract content and titles for already-captured sessions from the existing `archive/queue/**` payloads across all three agents, bypassing the importer's checksum de-duplication. It MUST support a dry-run mode and MUST be safe to re-run.

#### Scenario: Dry-run reports without writing
- **WHEN** backfill is run with `--dry-run`
- **THEN** it SHALL report the sessions it would re-extract without modifying any inbox or knowledge files

#### Scenario: Idempotent re-extraction
- **WHEN** backfill is run over the same queue payloads more than once
- **THEN** each run SHALL regenerate the same content/title without duplicating or corrupting records
