# Stage 3 研究：life-cycle 實作 — slash command、artifact contract、phase gating

## Executive Summary

Stage 3 的核心**不是發明新工作流**，而是把 `01.prompt-define-plan-build-verify-review-ship-resear.md` 裡已經談過的 `/research → /define → /plan → /build → /verify → /review → /ship` 工作流，**真正長進 Stage 1 的 PaulShiaBro daemon 與 Stage 2 的 memory home 裡**。否則它會停留在「一份很好的 README」，而不是「一條可追溯、可重現、可 gating 的 pipeline」。

三個最重要的判斷：

1. **slash command 是 presentation，不是真相來源**。真正的真相來源是 artifact（`docs/spec.md`、`reports/verify/*`、`experience/<slug>.md`），slash command 只是把「生成/更新某個 artifact + 註冊下一階段 gate」這件事給一個穩定入口。任何階段如果沒有 artifact 落地，它就沒有發生。
2. **每個階段都必須有 entry gate + exit gate**，而且 gate 判斷應該是**靜態可檢查**（schema、frontmatter、checksum、必填欄位），不是靠 agent 自我宣稱。這是 `agentrc` readiness check 思路的直接延伸。[^agentrc]
3. **life-cycle 不是單向管線，而是帶回退邊的 DAG**。`/verify` 失敗要能退回 `/build`、`/review` 抓到 doc drift 要能退回 `/plan`、`/ship` 被 `picoclaw-ops-companion` 擋下要能退回 `/review`；這些「回退邊」必須是一等公民，不能靠人工重打 prompt。

基於此，Stage 3 最適合的形態是：

- **Lifecycle runtime**：掛在 PaulShiaBro daemon 上的一個 subsystem，吃 slash command、派工到 coordinator、收齊 artifact 後 gate check、把結果寫回 inbox
- **Artifact contract 套件**：每個 phase 一份 schema（建議 JSON Schema + markdown template）
- **Gate engine**：純靜態檢查 + `session-health` + `problemmap` 三線組合
- **Lifecycle index**：每個 project 一份 `lifecycle.yaml`，記錄當前階段、上次 gate 結果、未完成 rework、版本 pin

## 2. 核心抽象：Lifecycle 是 artifact-driven，不是 prompt-driven

### 2.1 失敗模式

如果 Stage 3 以 prompt 為中心設計（「`/build` 就是發一個 build prompt」），會出現三種慢性失敗：

1. **Artifact drift**：agent 口頭說「我已經做了 X」，但 `docs/` 或 `reports/` 根本沒檔案，或欄位不符 schema
2. **Gate lost**：沒有任何靜態方式判斷當前階段是否已完成，導致 `/verify` 被跳過或重跑多次
3. **Rework 黑洞**：回退時只有人記得「我上次 plan 改了什麼」，memory 與 artifact 都沒留痕

### 2.2 替代設計：artifact-first + event-first

改成：

- `slash command → lifecycle runtime → emit event → worker produces artifact → gate check → update lifecycle state`
- 每個 phase 有三個事件：`requested`, `artifact_submitted`, `gate_passed`（或 `gate_failed`）
- 所有事件寫進 `~/.agents/memory/runtime/lifecycle/events.jsonl`，由 Stage 2 的 ledger 消費

### 2.3 為何 event-first 很重要

因為它同時給你：

- **可重放**：`events.jsonl` 本身就是 lifecycle truth，可以重建 lifecycle dashboard
- **可稽核**：每個 phase transition 都有 before/after、actor、artifact ref
- **可 debug**：`problemmap` 可以直接吃這份 log 做 route-first triage
- **可離線**：即使 PaulShiaBro 暫時掛掉，事件流仍是完整狀態

## 3. Slash command 設計

### 3.1 命令表（使用者可見的 surface）

| 命令 | 階段 | 預設行為 | 可選旗標 |
|---|---|---|---|
| `/research <slug>` | research | 建立/更新 `docs/research/<slug>.md`，派研究 worker | `--refresh` 強制重跑、`--from-issue <url>` 用 issue 當輸入 |
| `/define` | define | 從最近 research 聚合成 `docs/spec.md` | `--scope minor` 跳過 rollout 欄位 |
| `/plan` | plan | 產 `roadmap/test/task/todo/plan.md` | `--slice N` 只產當前切片 plan |
| `/build <task-id>` | build | 依 todo 派 bounded worker | `--parallel N`、`--branch <name>` |
| `/verify <scope>` | verify | 跑 test matrix + session-health | `--hw`、`--session-only`、`--replay <run-id>` |
| `/review <scope>` | review | code + doc drift review | `--skip-code`、`--skip-doc` |
| `/ship <scope>` | ship | commit / tag / push / deploy | `--dry-run`、`--skip-approval`（高風險，需 flag） |
| `/rewind <phase>` | 管理 | 退回指定 phase，artifact 版本打 tag | `--reason <string>` 必填 |
| `/status [slug]` | 管理 | 印出目前 lifecycle state、gate 結果、未完成 rework |  |
| `/hotfix <issue>` | 特殊 | 啟動 hotfix fast-path | 見 §7 |
| `/spike <slug>` | 特殊 | 啟動探索性短迴圈 | 見 §7 |

### 3.2 命令實作原則

1. **冪等**：同一 slug 的 `/research` 重複跑不應該產生重複檔案，應 merge / version
2. **可中斷**：ESC / `/cancel` 必須能安全停一個 worker，artifact 半成品移至 `runtime/lifecycle/partial/<run-id>/`
3. **權限**：`/ship` 必須經 `picoclaw-ops-companion` approval gate；`/rewind` 必須帶 `--reason`
4. **不靠記憶**：任何命令必要的 context（當前 spec 版本、上一次 gate 結果）都從 `lifecycle.yaml` 讀，不從 agent context 取
5. **單向 inbox 寫入**：每個命令完成時，把**一份副本 / 指標**寫進 `~/.agents/memory/inbox/<phase>/`，由 Stage 2 importer 整理

### 3.3 PaulShiaBro daemon 路由契約

```
Telegram / TUI / CLI
      │
      ▼
PaulShiaBro daemon
      ├── parse slash command
      ├── resolve project (cwd or --project)
      ├── load lifecycle.yaml
      ├── check entry gate for target phase
      │     └── fail → return gate report, do not dispatch
      ├── coordinator.create_job(phase, scope, artifacts_required, timeout)
      ├── emit event: <phase>.requested
      └── return job_id + stream handle
```

daemon 不自己執行 phase 邏輯，它只做 **dispatch + gate**，執行交給 coordinator 派給對應 persona worker（見 Stage 4）。

## 4. Artifact contract：每個 phase 的 schema

### 4.1 共通 frontmatter（所有 phase artifact 都要有）

```yaml
---
phase: research|define|plan|build|verify|review|ship
project: <project-slug>
slice_id: <uuid>          # 本輪 lifecycle 切片 id
artifact_kind: research|spec|roadmap|test|task|todo|plan|report|review|ship-record
version: <semver-ish>     # 該 artifact 在同一 slice 內的版次
supersedes: <prev-artifact-path>  # 若 rework
created_at: <ISO8601>
created_by: <persona-id>
source_session: <session-id>
gate_required: true|false
checksum: <sha256 of body>
---
```

Stage 2 的 provenance schema 已經有一部分（見 `02.obs-auto-moc` §3.3），Stage 3 要加上 `phase / slice_id / artifact_kind / supersedes` 四個欄位。

### 4.2 Phase-by-phase 必填欄位

| Phase | Artifact path | 必填 sections | Gate check（靜態） |
|---|---|---|---|
| research | `docs/research/<slug>.md` | `# 問題`、`# 現況盤點`、`# 限制`、`# 參考`、`# 建議方向`、`# 信心度` | 每節非空；參考至少 3 條；信心度枚舉值 |
| define | `docs/spec.md` | `# 目標與非目標`、`# 流程`、`# architecture 邊界`、`# failure model`、`# config/env contract`、`# 驗證方式`、`# rollout/rollback` | 所有 7 節必填；config/env 必須含完整變數表 |
| plan | `docs/roadmap.md` + `test.md` + `task.md` + `todo.md` + `plan.md` | 見 §4.3 | 五份文件齊全且 cross-ref 一致 |
| build | branch diff + `reports/build/<task-id>.md` | build summary、touched paths、unit test result、risks | 必須附 diff patch ref；允許路徑需對應 task.md |
| verify | `reports/verify/<run-id>/` bundle | test result、session-health、hw report（如適用） | 至少一線通過；任一線失敗必須附 failure classification |
| review | `reports/review/<run-id>.md` | code findings、doc drift findings、verdict | verdict 枚舉：`approve / changes_requested / block` |
| ship | `reports/ship/<version>.md` + commit + tag | commit sha、tag、changelog、deploy target、rollback plan | commit 有 conventional prefix；tag 符合 semver；rollback plan 非空 |

### 4.3 Plan phase 的五份文件分工（銜接 `01` §3.2 表格）

> 這裡只補 `01` 未寫到的 schema 層細節。

- `docs/roadmap.md`：每個 phase 必含 `id / goal / entry_conditions / exit_conditions / depends_on`
- `docs/test.md`：每個 test case 必含 `id / layer(unit|integration|e2e|hw|session|doc) / linked_spec_section / expected_artifact`
- `docs/task.md`：每個 task 必含 `id / linked_roadmap_phase / acceptance_criteria`
- `docs/todo.md`：每個 todo 必含 `id / linked_task / allowed_paths / estimated_effort / ready:true|false`
- `docs/plan.md`：只列當前 slice 要執行的 todo 子集 + 分派到哪個 persona

### 4.4 Artifact 存放雙軌

- **Versioned（進 git）**：`docs/*`、`reports/review/*`、`reports/ship/*`
- **Runtime（不進 git）**：`reports/verify/<run-id>/raw/*`、`runtime/lifecycle/partial/*`、`runtime/lifecycle/events.jsonl`

這條界線沿用 `gitagent` 的 versioned vs runtime 思路，也與 Stage 2 的 `runtime/` 目錄對齊。

## 5. Gate engine

### 5.1 Gate 三線並聯

```
entry_gate(phase) OR exit_gate(phase):
  ├── static_check     (frontmatter, schema, cross-ref, checksum)
  ├── health_check     (session-health score ≥ threshold, if applicable)
  └── route_check      (problemmap 未擋；若擋須明確 override)
```

三線中 `static_check` 必過；`health_check` 與 `route_check` 可依 phase 設定閾值或 `optional`。

### 5.2 Gate 失敗後的預設行為

- 產出 `runtime/lifecycle/gates/<slice>/<phase>/<timestamp>.md` gate report
- lifecycle state 改為 `blocked:<phase>`
- 通知 Telegram + TUI
- 不自動派工；等人下 `/rewind <phase>` 或 `/status` 確認後手動重跑

> **Do not** 讓 daemon 自動重試直到通過。Auto-retry 是 silent failure 的最大來源。

### 5.3 Gate 可 override，但 override 本身是事件

```
/gate override --phase verify --slice <id> --reason "hw 暫無"
→ events.jsonl: {kind: "gate.override", phase, slice, reason, actor, ts}
→ lifecycle.yaml 下個 ship 自動把這筆 override 列入 release note
```

override 不刪除原 gate report，而是附加事件；審計時要能看到「誰在什麼時候為什麼繞過」。

### 5.4 與 `agentrc` 的關係

`agentrc` 的 readiness / drift / eval 思路可以直接變成**跨 slice 的背景 gate**：

- readiness：`AGENTS.md`、`projects.yaml`、hooks 齊不齊
- drift：`docs/spec.md` 與 `README.md` / code API 的對照
- eval：把 `test.md` 的 case 當 eval 題庫

這些檢查不與單一 slash command 綁，而是週期性跑，結果貼到 `runtime/lifecycle/health/agentrc/`。[^agentrc]

## 6. 回退邊：rework / rewind / escalate

### 6.1 三種回退事件

| 事件 | 觸發者 | 語意 | 影響 |
|---|---|---|---|
| `rework` | reviewer / tester | 目前 artifact 需修，但 phase 不變 | artifact version++；不重置下游 |
| `rewind` | manager / user | 退回更早 phase | 下游 artifact 全部標 `stale`，新版從退回點重開 |
| `escalate` | 任何 persona | 遇到超出 scope 的阻塞 | 通知 manager + user；lifecycle 暫停 |

### 6.2 rewind 的語意

假設目前在 `/review`，執行 `/rewind plan --reason "測試矩陣漏掉 hw 場景"`：

1. 當前 slice 分岔：`<slice-id>` → `<slice-id>-r1`
2. `plan` 階段之後所有 artifact 標記 `status: stale, superseded_by: <new-slice>`
3. `/plan` 在新 slice 上重跑；下游 phase 的 entry gate 會發現 plan artifact 版次變了，觸發重跑
4. Stage 2 memory 照收這些 stale artifact，但 classifier 應標記為「歷史版本」，不進 canonical work-centric

### 6.3 為何要分岔而不是直接覆蓋

因為 `experience/` 要保留「為何要 rewind」的真相。如果直接覆蓋，幾次以後就會退化成「這個專案好像改過什麼，但沒人記得」。

## 7. 特殊 lifecycle：hotfix、spike、doc-only

### 7.1 Hotfix fast-path

觸發：`/hotfix <issue>`

- 跳過 `research`、`define`、`plan`
- 要求：必須在 `docs/spec.md` 已存在的 scope 內（用 spec section ref 驗證）
- 流程：`/build → /verify → /review → /ship`，但 `/verify` 與 `/review` 的 gate 閾值可升級（更嚴）
- 產出：`experience/hotfix-<issue>.md` 含 root cause、fix、lessons
- Ship 後必須在下一輪常規 lifecycle 補 `define` 更新，否則 agentrc drift gate 會報警

### 7.2 Spike / RFD / PoC

觸發：`/spike <slug>`

- 建 `docs/research/<slug>-spike.md`，但標 `artifact_kind: spike`
- 不產 spec，不進 build
- 有明確 expiry（預設 14 天）；過期自動 faceout
- 結果要不升級成正式 research → /research fork，要不寫入 `experience/rejected-ideas.md`
- **不允許** spike 直接改 code；若需要 code，必須走正式 lifecycle

### 7.3 Doc-only change

觸發：`/plan --doc-only` 或直接 PR

- 跳過 `/build`、`/verify` 程式線
- 但 `/verify` 的 doc 線仍要跑：README / spec drift 檢查
- Ship 走 `docs:` conventional commit prefix
- 主要目的：把 `review` 的 doc drift finding 變成可 ship 修復

## 8. `/ship` 的合約：version、release、rollback

### 8.1 Ship 最小合約（補 `01` §3.2 G）

- commit：conventional-commit，帶 `slice_id` trailer
- tag：semver；pre-release 用 `-rc.N` 後綴
- changelog：`reports/ship/<version>.md` 自動從事件流生成（research/define/plan/build/verify/review 的主事件）
- deploy target：宣告部署目標；不部署也要寫 `target: none` 保留欄位
- rollback plan：強制欄位；最少一句話說明如何回退

### 8.2 Version 策略建議

- Stage 1 / Stage 2 服務本體走 semver
- 每個 downstream 專案（被 workflow 管的專案）可自選；lifecycle runtime 只要求「tag 存在且唯一」
- lifecycle.yaml 同時 pin 一個 `workflow_version`，用來限制哪些 slash command 可用

### 8.3 Rollback 的三層

| 層 | 範圍 | 工具 |
|---|---|---|
| commit 層 | revert commit | `git revert` + 新 slice 做 `/ship` |
| deploy 層 | 退回上一版部署 | `picoclaw-ops-companion` 的 approval-guarded revert |
| data 層 | schema / memory 退回 | Stage 2 backup / snapshot（見 §12 的盲點清單） |

Rollback 不是 `/rewind` 的同義詞：rewind 處理「這個 slice 的歷史」，rollback 處理「已經 ship 出去的效果」。

## 9. Hook 佈線

### 9.1 標準 hook 清單

| Hook | 觸發點 | 預設動作 |
|---|---|---|
| `session.end` | Claude Code / Copilot / Codex session 結束 | codex-lesson 提煉 → `inbox/sessions/<session-id>.md` |
| `session.pre_compact` | context compact 前 | 把當前 lifecycle state snapshot 寫 `inbox/compact/<session>.md` |
| `lifecycle.phase.requested` | slash command 派工前 | 在 TUI / Telegram 播報 |
| `lifecycle.gate.failed` | gate 檢查失敗 | 通知 + 寫 inbox |
| `lifecycle.slice.closed` | `/ship` 成功 | 把整 slice 的 artifacts 歸檔 `work-centric/<project>/experience/` |
| `cron.daily` | 每日 | agentrc readiness / drift check |
| `cron.weekly` | 每週 | codex-project-insights 聚合 |
| `git.pre_push` | push 前 | 確認 slice 有 ship record 對應 |

### 9.2 Hook 配置收斂

所有 hook 寫在一份 `config/lifecycle-hooks.yaml`，**不要散在各 agent 的 config**。agent 啟動時透過 env 或 wrapper 把這份 hook 表接上。

### 9.3 Hook 失敗策略

- 預設 non-blocking（hook 失敗不擋 slash command 主流程）
- 但失敗會寫進 `runtime/lifecycle/hook-failures.jsonl`；超過閾值由 `wake-up` 提報

## 10. 觀測與 lifecycle dashboard

### 10.1 必要 metric

- 每個 project 當前 phase
- 每個 phase 的累積時間（running avg / p95）
- gate 失敗率（per phase）
- rework 次數 / rewind 次數（per slice）
- hotfix 比例
- ship 成功率
- stale spike 數量

### 10.2 Dashboard 策略

沿用 `01` §5.3 的 federated 思路，**不做新 web dashboard**：

- live 狀態：`/status` CLI + TUI 右側欄
- 品質：`session-health` HTML + `reports/verify/*/index.html`
- 長期：Obsidian overlay + Stage 2 wake-up bundle

### 10.3 Lifecycle index 檔案

`~/.agents/memory/work-centric/<project>/lifecycle.yaml`：

```yaml
project: paulshaclaw
current_slice: slice-2026-04-20-a
current_phase: verify
workflow_version: 0.3.0
last_ship: v0.2.1
open_rework: []
open_rewind: []
stale_spikes: [ui-acp-exp]
gates:
  verify:
    last_check: 2026-04-20T11:03Z
    status: running
  review:
    last_check: 2026-04-19T22:10Z
    status: passed
```

這份檔是 lifecycle runtime 的主要 state，其他 artifact 只是證據。

## 11. 與 Stage 1、Stage 2、Stage 4 的介面

### 11.1 與 Stage 1（PaulShiaBro + coordinator）

- PaulShiaBro daemon 提供 slash command 入口；不自己實作 phase 邏輯
- coordinator 負責派 job；每個 phase 的 `required_artifacts` 成為 job 完成條件
- job 完成時由 daemon 收 artifact → 寫 lifecycle event → 跑 gate
- TUI 的右欄接 lifecycle index + gate report

### 11.2 與 Stage 2（memory home + dream service）

- 每個 phase 結束把**產出副本 / pointer** 寫 `inbox/<phase>/`
- inbox classifier 依 `artifact_kind` 路由：spec → `work-centric/<project>/plan/`、research → `research/`、verify/review → `experience/`、ship → `experience/ship/`
- dream service 把 `experience/` 中高穩定內容原子化到 `knowledge/incidents/` 或 `knowledge/methods/`
- faceout state 機制接管 stale spike、過期 hotfix note

### 11.3 與 Stage 4（persona / 角色分工）

- 每個 phase 都綁一組 allowed personas（見 Stage 4 研究）
- Manager persona 可下所有 slash command；其他 persona 只能在自己 phase 範圍內產 artifact
- `/rewind` 限定 Manager persona 或 user
- Shipper persona 不能跑 `/build`；Reviewer persona 不能改 code

### 11.4 與 Stage 1 branding（logo）

- 每次 `/ship` 的 release note 頭部可嵌入 lobster banner B variant
- `/status` TUI 左上角顯示 U+E100 + 當前 phase 縮寫
- changelog 生成時把 banner C 以註解形式保留，方便 copy 到 social post

---

## 12. 治理盲點（目前 Stage 3 還沒完全想清楚的）

1. **workflow 版本遷移**：當 `docs/spec.md` schema 改了（加新 section），舊 slice 的 artifact 是否要回填？建議做 `workflow migration notes`，不強制回填。
2. **多人 / 多 agent 同時在同一 slice 工作**：slice 粒度是否該再細切？或引入 sub-slice？
3. **跨專案 lifecycle**：一個改動同時影響多個 repo 時，lifecycle index 放哪？建議放 meta repo 並以 `projects.yaml` 引用子專案。
4. **Artifact 大附件**：verify bundle 若帶大 log（serial trace、HAR），inbox 會爆。需要 pointer-only + hash 機制。
5. **離線模式**：daemon 掉時，slash command 能否先落 artifact 事後 sync？短期不建議做；保留 inbox fallback 寫入即可。
6. **多語言 artifact**：使用者寫 zh-TW，部分 tool 輸出 en；spec/plan drift check 要能跨語言。短期：只檢 section 存在 + 必填 key，語言獨立。
7. **Lifecycle 自身的測試**：slash command / gate engine 本身需要回歸測試。建議每個 phase 準備 golden slice 範例，CI 上跑「把範例走一輪」確認 gate 行為不退化。
8. **Gate check runtime 成本**：若每個 artifact 都跑 full session-health，`/status` 會很慢。建議 cache + 僅在 artifact checksum 改時重算。
9. **AGENTS.md 擴散**：多 repo 各有 AGENTS.md 容易漂移；應由 Stage 3 lifecycle runtime 擁有一份 canonical `workflow-agents.md`，各 repo 以 `include` 方式引用。

---

## 13. 建議的落地分階段

### Phase 3.0 — Artifact schema 與 lifecycle index（先有真相）

- 定義 frontmatter schema（phase / slice_id / artifact_kind / supersedes / checksum）
- 建 `lifecycle.yaml` template
- 寫一組 static gate check script（pure python / bash，無 daemon 依賴）
- 手動跑 lifecycle，用人工呼叫 gate script

完成標誌：任何專案可以在不啟動 daemon 的情況下，把 `docs/spec.md` / `reports/*` 放上並通過 gate check。

### Phase 3.1 — Slash command 路由（接上 PaulShiaBro）

- 把 `/research`、`/define`、`/plan`、`/status`、`/rewind` 接進 PaulShiaBro
- 只做 dispatch → coordinator，artifact gate 沿用 Phase 3.0 script
- `/build`、`/verify` 仍可手動；先確認 routing 穩

完成標誌：`/status` 可以看當前 phase、gate 結果、上一個 ship。

### Phase 3.2 — Gate engine 與事件流

- `events.jsonl` 落地
- gate engine 三線並聯
- hook 佈線（session.end、gate.failed、slice.closed）
- Stage 2 importer 認得 phase frontmatter

### Phase 3.3 — 回退、hotfix、spike

- `/rewind` 分岔語意
- hotfix / spike fast-path
- 特殊 lifecycle 的 gate 閾值調整

### Phase 3.4 — Release / rollback / deploy

- `/ship` 綁 conventional-commit、tag、changelog
- 接 `picoclaw-ops-companion` approval
- rollback 三層合約

### Phase 3.5 — Observability & drift

- lifecycle dashboard 視圖
- agentrc drift gate 週期化
- golden slice 回歸測試

---

## 14. 不建議現在就做的事

1. **不要做 workflow 本身的 DSL**。YAML + frontmatter 已足夠；發明 DSL 會把治理成本拉高。
2. **不要把 lifecycle 做成單機限定**。從第一版起 artifact 就應可序列化跨機（但不急著做分散式）。
3. **不要讓 gate engine 自動改 artifact**。gate 是 observer，不是 mutator。
4. **不要把每個階段都做成 agent-only**。Operator 手動補 artifact 應該是一等路徑；lifecycle 只需確認 artifact 合規。
5. **不要把 Stage 2 的 faceout 與 Stage 3 的 slice close 混在一起**。前者是記憶衰減，後者是 lifecycle 關閉；生命週期不同。

---

## 15. 結論

Stage 3 最有價值的產出不是「一套新工具」，而是**把 lifecycle 定義成「可以 grep、可以 diff、可以 replay」的檔案真相**。只要這層到位：

- Stage 1 只需承擔 dispatch 與視覺化
- Stage 2 只需承擔記憶整理
- Stage 4 只需承擔角色綁定

在落地順序上，**先寫 schema + gate script**，再接 PaulShiaBro，再做 hook / observability，最後再談 rollback / drift。這條路可以讓你在每一步都有可用的東西，而不是等整套做完才第一次看到成果。

## 16. Confidence assessment

### 高信心

- artifact-first / event-first 比 prompt-first 更適合長期維運
- 靜態 schema gate 是最便宜又最有效的一層
- hook 佈線應集中到 `config/lifecycle-hooks.yaml`
- `/rewind` 必須分岔而不是覆蓋

### 中信心

- workflow 版本遷移策略（是否強制回填舊 artifact）
- gate 三線閾值的預設值
- changelog 自動生成的欄位取捨

### 低信心 / 待驗證

- 跨專案 lifecycle（meta repo 邊界）
- 多語言 artifact 的 drift check 做到什麼程度才夠
- lifecycle 本身的回歸測試要怎麼寫才不會跟 `test-playbook` 重疊
