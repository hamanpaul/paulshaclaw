# `obs-auto-moc` 第二階段記憶管理中樞研究

## Executive summary

你的第二階段方向是對的，但**不建議把它做成另一套大而全的「GBrain/MemPalace 替代品」**。比較穩的路線是：

1. **保留 `obs-auto-moc` 的強項**：deterministic intake、memory ledger、wake-up/dream/persona/synthesis overlays、proposal-first。
2. **把 memory home 從 Obsidian vault 抽離**到 `~/.agents/memory`，讓 Obsidian 從「唯一資料根」退為「可選閱讀介面／知識可視化介面」。
3. **把 dream mode 拆成獨立 janitor/orchestrator service**，不要只掛在 pipeline 後面順手跑。
4. **把 raw / distilled / canonical / indexed 四層分離**，避免 session 雜訊直接污染知識層。
5. **先做 lexical + relation retrieval**，embedding / vector / graph cache 放後面；否則很容易先把維運複雜度做爆。

結論上，這個階段最適合採用：

- **`obs-auto-moc`** 當 canonical memory compiler
- **`agent-skills`** 借 phase mapping
- **`agentrc`** 借 readiness / eval / drift gate
- **`gitagent`** 借 repo layout 與 versioned-vs-runtime 邊界
- **`graphify`** 借 code graph cache / watch / MCP 觀念
- **`symphony`** 借 orchestration boundary
- **`gbrain` / `mempalace`** 只借少數觀念，不直接整套移植

## 一、現有 `obs-auto-moc` 哪些可以直接沿用

### 1. 已經存在的能力

本機 `obs-auto-moc` 已經不是單純的 MOC 更新器，它其實已經有一個輕量 memory substrate：

- `memory_ledger.py`
  - `entities.jsonl`
  - `relations.jsonl`
  - `events.jsonl`
  - `citations.jsonl`
  - `note-facts.jsonl`
  - `contradictions.jsonl`
- `wakeup.py`
  - 產生 wake-up bundle
- `dream.py`
  - 產生 consolidation proposals
- `persona_goal.py`
  - 蒸餾 heuristics / goals
- `synthesis.py`
  - 產生 `VERIFIED / SYNTHESIZED / OPEN QUESTION`
- `session_insights.py`
  - ingest distilled session artifacts，而不是 raw session

### 2. 最有價值的既有設計

這幾個設計非常值得保留：

- **canonical note contract**
  - `Compiled Truth`
  - `Timeline`
  - `Sources`
  - `Related`
- **secondary ledger**
  - canonical note 之外有可查詢的 side index
- **proposal-first**
  - dream / persona / synthesis 先落 overlay，不直接回寫 canonical
- **distilled-artifact-first**
  - `import-session-insights` 已經證明 ingest distilled outputs 可行

### 3. 目前最大的結構限制

它現在仍有三個硬限制：

1. **memory root 仍附著在 vault**
   - 預設是 `vault/claw/moc/...`
2. **allowed note scope 仍綁死**
   - `root-note / TechVault / WorkVault / PersonalVault`
3. **dream 仍是命令，不是長駐服務**
   - 現有 systemd scaffold 更像 pipeline timer，不是 memory janitor

所以第二階段的核心，不是重寫 memory feature，而是**把資料根、服務邊界、檢索邊界抽象化**。

## 二、建議的第二階段目標架構

### 1. 核心原則

- **memory home-first**：所有 agent memory 先進 `~/.agents/memory`
- **obsidian-optional**：Obsidian 不再是唯一資料根，而是可治理介面
- **compiler model**：`obs-auto-moc` 是 compiler / janitor，不是人工筆記本本體
- **raw 與 distilled 分層**
- **service 與 command 分層**
- **versioned 與 runtime 分層**

### 2. 建議目錄

### 3. 四層記憶定位

| 層級 | 建議位置 | 性質 | 是否可直接給 RAG |
|---|---|---|---|
| 短期記憶 | `inbox/` | 原始、未整理、任務中間產物 | 不建議 |
| 中期記憶 | `work-centric/<project>/` | 專案上下文與工作真相 | 可以，需過濾 |
| 長期守則 | `work-centric/common-sense/` | 穩定行事準則與偏好 | 可以 |
| 知識沉澱 | `knowledge/` | 原子化、可重組、可關聯 | 最適合 |

### 4. 新的資料流

```text
agent session
  -> hooks / wrappers
  -> ~/.agents/memory/inbox
  -> importer / classifier
  -> work-centric or common-sense
  -> atomizer / linker
  -> knowledge
  -> ledger / health / wake-up / dream / synthesis overlays
```

## 三、你目前設計裡最需要補的點

這一段是本次研究最重要的結論：**你大的方向幾乎都想到了，但有幾個關鍵「治理層」還沒被明確定義。**

### 1. Project identity 與歸檔規則

你現在有 `{{project name}}`，但還缺：

- project slug 規則
- repo path 對應
- remote URL 對應
- alias 對應
- monorepo / multi-repo / subproject 歸屬

建議新增 `projects.yaml`：

```yaml
projects:
  obs-auto-moc:
    slug: obs-auto-moc
    roots:
      - /home/paul_chen/prj_pri/custom-claw-tools/obs-auto-moc
    remotes:
      - github.com/hamanpaul/obs-auto-moc
    aliases: [auto-moc, obs-moc]
```

否則 inbox classifier 在多專案場景下很容易誤歸檔。

### 2. 多 agent 併發寫入治理

你目前規劃了很多 agent 同時工作，但還沒定義：

- 同一專案的檔案誰能寫
- 同一篇 MOC 誰能更新
- 何時 append，何時 rewrite
- dream mode 跟 agent session 同時改檔時怎麼避衝突

至少要有：

- file lock / advisory lock
- append-only inbox
- work-centric 的 single-writer policy
- knowledge atomization 的 idempotent key

否則夢境整理和 agent 收尾 hook 很容易互踩。

### 3. Provenance / citation / traceability contract

你有強調 distilled artifacts，但還缺一個一致欄位：

- `source_agent`
- `source_session`
- `source_repo`
- `source_commit`
- `source_artifact`
- `captured_at`
- `confidence`
- `distilled_from`

建議任何進入 `work-centric` / `knowledge` 的檔案 frontmatter 至少有：

```yaml
memory_layer: inbox|work|knowledge
project: obs-auto-moc
source_agent: copilot-cli
source_session: abba4d80-...
source_artifact: plan|research|lesson|insight|report|trace
captured_at: 2026-04-19T...
provenance:
  repo: ...
  commit: ...
  path: ...
```

### 4. Faceout 不是刪除，而是 state machine

你提到 faceout，但還沒定義 transition：

- active
- cooling
- faceout
- archived
- reactivated
- excluded

以及觸發條件：

- N 天未引用
- 與 canonical 內容衝突
- 來源失效
- 專案結束
- 被新知識取代

建議 faceout 改為 metadata 狀態，而不是移檔或刪除。

### 5. inbox retention / garbage collection

你強制所有 agent 把東西丟 inbox，這是對的，但一定會爆量。

還缺：

- inbox 最大停留時間
- 重複內容偵測
- 大檔附件策略
- OCR / transcript / binary sidecar 管理
- 失敗重試與死信區

建議：

- `inbox/queue/ready|failed|dead-letter`
- 大附件只存 pointer + hash
- raw transcript 不直接進 knowledge

### 6. 安全邊界與機敏資料

如果 memory 中樞 24x7 跑，會自然吸進：

- access tokens
- cookies
- internal URLs
- private meeting notes
- screenshots with secrets

需要在 importer 前加：

- secret redaction
- classification (`public/internal/confidential/restricted`)
- source allowlist / denylist
- external publish guard

這一層在你的設計裡目前還太淡。

### 7. Search 與 RAG 不應直接綁死在同一 backend

你想把 `knowledge` 當 RAG 主來源，方向正確；但設計上應分三層：

1. **canonical store**
   - markdown / yaml / jsonl
2. **retrieval index**
   - FTS / sqlite / vector / graph cache
3. **reasoning surface**
   - wake-up / synthesis / agent context loader

不要把 `knowledge` 直接等同於 vector DB。  
先做：

- file metadata index
- lexical search
- relation traversal
- recency / citation weighting

embedding 是後加項。

### 8. Agent instruction propagation

你提到要在 `AGENTS.md / INSTRUCTION.md / CLAUDE.md / GEMINI.md` 中強制 inbox 寫入。  
但本機現況是：

- 有 `AGENTS.md`
- 沒找到穩定在用的 `CLAUDE.md / GEMINI.md / INSTRUCTIONS.md`

所以第一版不要假設「每種 agent 都吃 instruction file」。

建議採三層：

1. **shared policy**
   - `~/.agents/AGENTS.md`
2. **wrapper env**
   - 啟動各 CLI 時注入 `AGENT_MEMORY_HOME`, `AGENT_INBOX`, `AGENT_PROJECT`
3. **per-tool adapters**
   - 能吃 instruction file 的才補對應檔案

### 9. 觀測與自我修復

24x7 service 若沒有 health loop，很快就變成 silent failure。

至少需要：

- queue depth
- last successful dream cycle
- import failure count
- lock contention
- faceout candidates pending
- stale project MOC count
- index lag

這些要能被 `wake-up` 看到，而不是只存在 systemd log。

## 四、外部方案對照與適配建議

## 4.1 `agent-skills`

### 可借

- `define / plan / build / verify / review / ship` phase mapping
- hooks 與 slash command 心智模型
- skill-based workflow decomposition

### 適合你的原因

你現在要的其實不是單一 agent，而是**life-cycle 可編排化**。  
`agent-skills` 最值得借的不是 skill 內容本身，而是：

- phase vocabulary
- phase-to-artifact mapping
- hooks 入口
- `/plan` `/build` `/review` `/ship` 的心智模型

### 不建議直接照搬

- 全量文檔與命令命名
- 以單一工具生態為前提的路徑

### 建議用法

把它變成你的 lifecycle spine：

| Phase | 你的 memory 寫入點 |
|---|---|
| define / research | `inbox/research` -> `work-centric/<project>/research` |
| plan | `inbox/plans` -> `work-centric/<project>/plan` |
| build / verify / review | `inbox/reports` -> `experience/` |
| ship | `experience/` + `common-sense/` 回寫規則 |

## 4.2 `agentrc`

### 可借

- readiness 檢查
- eval cases
- drift-in-CI
- policy file

### 為何重要

你的系統未來會越來越多：

- shared instructions
- wrappers
- hooks
- project memory policy

這些最怕**漂移**。`agentrc` 的價值不是幫你生成 instruction，而是幫你做：

- 「專案是否 AI-ready」的靜態檢查
- instruction / workflow 品質退化時 fail CI

### 建議導入點

- 檢查 `AGENTS.md` / workflow / hooks 是否存在
- 檢查 `projects.yaml` / `memory.yaml` 是否有效
- 評估 dream / importer / instruction wrappers 是否齊

### 結論

**非常值得借，而且要早借。**  
這會是你 memory 中樞的治理閘門，不是記憶引擎本身。

## 4.3 `graphify`

### 可借

- code graph cache
- `watch` + `serve`
- export / report / MCP
- plain-file pipeline、低共享狀態

### 為何適合

你之後一定會遇到「專案記憶有文字知識，但缺 code topology」的問題。  
`graphify` 的定位很適合做：

- repo code graph sidecar
- per-project architecture snapshot
- always-on summary input

### 不建議直接借的部分

- 不要把 graphify 當你的通用 memory substrate
- 不要把所有筆記／經驗資料硬塞成 graph node

### 建議位置

每個專案額外生成：

```text
work-centric/<project>/architecture/
runtime/indexes/code-graph/<project>/
```

然後讓 wake-up / synthesis 在需要時引用。

### 結論

**適合當「專案代碼記憶加速層」**，不是主記憶庫。

## 4.5 `gbrain`

### 可借

- thin harness / fat skills
- signal detector / brain-first lookup
- nightly maintain / dream cycle
- deterministic job queue 與 sub-agent 分流

### 不建議直接搬

- 它太像一整套 opinionated product
- DB / pgvector / job system / skillpack 綁得很深
- 對你現有 `obs-auto-moc + notes + local files` 會太重

### 最值得借的三件事

1. **brain-first lookup**
   - agent 回答前先查本地記憶
2. **signal detector always-on**
   - 每則訊息都做輕量記憶抽取
3. **deterministic work vs judgment work 分流**
   - 定型任務走 job / minion
   - 需要判斷的才走 reasoning agent

### 結論

**借運營哲學，不借整套技術棧。**

## 4.6 `mempalace`

### 可借

- local-first
- verbatim storage
- pluggable backend contract
- hooks: save / pre-compact
- agent diaries / per-agent wings

### 為何值得參考

你要集中 agent memory，而 MemPalace 明確區分：

- 原始對話保留
- semantic retrieval
- backend abstraction
- compact 前 hook

這對你設計 inbox 與 raw retention 很有幫助。

### 不建議直接照搬

- palace / wing / room / drawer 的隱喻模型
- 直接把所有 memory 都 vector 化

### 最值得吸收

1. **verbatim raw archive 不做摘要取代**
2. **pre-compact hook**
   - 很適合接你的 session end / context compact hook
3. **backend interface**
   - 幫你保留未來換 FTS / vector backend 的彈性

### 結論

**很適合當 raw memory 與 retrieval backend 抽象的參考。**

## 4.7 `symphony`

### 可借

- 長駐 orchestrator 邊界
- `WORKFLOW.md` repo-owned policy
- per-issue workspace
- runner / orchestrator / config / integration / observability 分層

### 為何重要

你未來第一階段主核心 + TUI + task manager，其實很容易往這個方向走。  
`symphony` 給你的價值是：

- scheduler 是 scheduler
- ticket / PR 寫回邏輯不強耦合在 orchestrator
- runtime state 與 workflow policy 分層

### 對第二階段的直接啟發

你的 dream service / task coordinator 也應分層：

- policy layer
  - memory workflow rules
- config layer
  - path/env/project config
- coordination layer
  - queue / retry / concurrency
- execution layer
  - importer / classifier / atomizer / faceout / index refresh
- observability layer
  - health / logs / dashboard

### 結論

**非常值得借 boundary，不一定要借實作。**
