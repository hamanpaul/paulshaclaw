* work-centric: 工作記憶，此處文件以專案為導向，不做原子化拆分，盡可能完整保留檔案。

*** research: 對應life-cycle的`/research` 指令，為專案研究檔，用於概念模糊、問題空間不清楚、要盤點既有資產與外部參考時；有可能不只一份專案研究檔，盡可能完整保留。
*** story: 為專案的user story, 主要由inbox資料拼湊而成，需整理成符合 INVEST原則，重要!!!**需隨專案進度==同步==更新**。
*** plan: 對應life-cycle的`/define` `/plan` 指令, 存放專案計劃書、規格書、roadmap、testplan、tasks、todos ，重要!!!**需隨專案進度==同步==更新**。
*** experience: 對應life-cycle的`/build` `/verify` `/review`指令，主要用來記錄開發或除錯過程中遇到的問題及解決方式，
	```
	# {{project name}} 問題事件狀態 (日期，專案版號，commit hash)
	- **Remote repo**: 如無則記錄本路專案repo路徑
	- **Error**: 問題狀態描述，例如 `更新todo list 失敗`
	- **Root cause**: 問題根因，例如 `專案無修改權限`
	- **Solution**: 
	- ** Serious score**: EMERG / ALERT / CRIT / ERR / WARNING / NOTICE / DEBUG
	```
	嚴重等級說明如下:
	EMERG  /* systemis unusable */
	ALERT /* actionmust be taken immediately */
	CRIT /*critical conditions */
	ERR /* errorconditions */
	WARNING /* warning conditions */
	NOTICE /* normalbut significant */
	INFO /*informational */
	DEBUG /*debug-level messages */

以知識層級來說，inbox對應短期記憶(一級)，為每個task/agent在執行過程中需要維護的記憶區，於task/agent的life-cycle結束後整理。
work-centric/{{project name}} 對應為中期記憶(二級，Memorize)，為task/agent在啟動後需要先至此瞭解專案目前的目的、架構、進度，然後提取至inbox維護。
work-centric/common-sense 對應為長期記憶(三級，Understand)，為task/agent在啟動後的行事準則，預期對應life-cycle 的Persona其使用者習慣。
knowledge 則為最終的知識沉澱池(四級，內化層級，Expand & Active)，也需做為 RAG語意檢索的主要資料來源。
`obs-auto-moc` 作為 Obsidian memory substrate，吃 `codex-lesson` / `codex-project-insights` 產出的 distilled artifacts，而不是直接 ingest 原始 session。
