# 適合 hamanpaul 的 define → plan → build → verify → review → ship 工作流研究

## 結論先行

最適合你現在資產狀態的，不是「先做一個全新大一統 agent 平台」，而是把你已經有的能力收斂成 **四層式工作流**：

1. **控制層**：`Max` + `coordinator` + `smux-manager-skill`，負責接收需求、切 task、派工、收斂結果。[^max][^coordinator][^smuxmgr]
2. **知識層**：`obs-auto-moc` 作為 Obsidian memory substrate，吃 `codex-lesson` / `codex-project-insights` 產出的 distilled artifacts，而不是直接 ingest 原始 session。[^obsmoc-readme][^codex-lesson][^codex-insights]
3. **驗證層**：一般 repo 沿用原生 test/build；agent session 品質用 `session-health`；硬體/嵌入式流程用 `TestPilot + serialwrap`。[^test-playbook][^session-health][^testpilot][^serialwrap]
4. **治理層**：`problemmap` 做 route-first failure diagnosis，`conventional-commit` + guarded push 做 ship，`picoclaw-ops-companion` 承接高風險 approval / audit。[^problemmap][^convcommit][^opscomp]

核心策略是 **hub-and-spoke，不是 full mesh**。也就是：**manager 是唯一 task authority，worker 只負責 bounded execution 與 artifact 回傳**。這個方向同時符合你現有的 `smux-manager-skill`、`coordinator`、`picoclaw-ops-companion` 設計，也能避免 mesh 失控、上下文漂移與權限外溢。[^smuxmgr][^coordinator][^opscomp]

### 2.1 為什麼是 hub-and-spoke，而不是 mesh

`smux-manager-skill` 明確規定 manager 是唯一 task authority，worker 只接 `TASK_ASSIGN / RESULT / FAIL / ASK`，而且建議把長輸出落 artifact 檔，不要直接把 pane 當資料庫。`coordinator` 也明確限制 relay 只能在 job 內已宣告 agent 之間傳送，且 job scope 衝突時要拒絕。[^smuxmgr][^coordinator]

所以你的 **MVP 不應該做「每個 agent 都能互相直接改別人的任務」的 full mesh**。建議改成：

- **manager 持有唯一 task graph**
- **worker 持有自己當前 task envelope**
- **橫向溝通要經 manager relay**
- **長輸出永遠寫 artifact，再回傳 path**

這樣才符合你現有工具的強項，也比較容易做審計、回放、重試與 ProblemMap 診斷。[^smuxmgr][^problemmap]

### 2.2 你的控制平面應該拆成三個層次

| 層次 | 推薦主件 | 職責 |
| --- | --- | --- |
| **互動入口** | `Max` | 你從 Telegram / TUI 發話、查狀態、取消工作、切模型 |
| **任務編排** | `coordinator` + `smux-manager-skill` | 把 spec / todo 拆成 bounded jobs 與 pane workers |
| **高風險執行** | `picoclaw-ops-companion` | 需要 approval / audit / policy 的 push、install、remote op |

這三層不要混成一個 daemon。`Max` 適合當 always-on orchestrator；`coordinator` 適合做可觀測、可 replay 的 job；`picoclaw-ops-companion` 適合承接高風險批准流程。[^max][^coordinator][^opscomp]

## 3. 建議的 lifecycle：/research → /define → /plan → /build → /verify → /review → /ship

### 3.1 使用者看到的命令語意

我建議你保留你自己的 `/research`，但把「define」實作成與 `agent-skills` 的 `/spec` 同義。也就是：

| 使用者命令       | 內部語意                                             | 主要輸出                                                                              |
| ----------- | ------------------------------------------------ | --------------------------------------------------------------------------------- |
| `/research` | 模糊概念探索、內外部資料盤點、風險前置研究                            | `docs/research/<slug>.md`                                                         |
| `/define`   | 把研究收斂成 spec（可對應 `/spec`）                         | `docs/spec.md`                                                                    |
| `/plan`     | 把 spec 轉成 roadmap / tests / tasks / todos        | `docs/roadmap.md`, `docs/test.md`, `docs/plan.md`, `docs/task.md`, `docs/todo.md` |
| `/build`    | 依 ready todos 開平行 worker 做 bounded slices        | branch diff, artifacts, implementation notes                                      |
| `/verify`   | 執行 repo tests + scenario/hardware/session checks | test reports, logs, session score                                                 |
| `/review`   | code/doc/report review 與修補建議                     | review report, issue list                                                         |
| `/ship`     | commit / tag / push / release / deploy           | commit, tag, release note, deploy record                                          |

這個分法本質上沿用 `agent-skills` 的 DEFINE / PLAN / BUILD / VERIFY / REVIEW / SHIP lifecycle，但前面補上一層你的 `/research`。[^agent-skills]

### 3.2 每一階段的最佳實作

#### A. `/research`

**用途**：概念模糊、問題空間不清楚、要盤點既有資產與外部參考時。

**主要執行者**：

- `skill-code-tracker` / LSP / `graphify`：查 code 結構與跨檔關聯。[^graphify]
- `problemmap`：如果研究起點其實是「為什麼之前 workflow 失敗」。[^problemmap]
- `obsidian-cli` / `obs-auto-moc query-memory`：回讀你既有筆記與 memory overlays。[^obsmoc-readme]

**產出**：

- `docs/research/<slug>.md`
- 問題陳述、現況盤點、限制、參考 repos / skills、建議方向

**退出條件**：

- 已能明確說出「做什麼 / 不做什麼 / 成功長什麼樣」

#### B. `/define`

**用途**：把 research 轉成單一 canonical spec。

**主要執行者**：

- `agent-skills` 的 spec-first 思維
- doc workflow（你的 `docs/spec.md`）

**產出建議**：

`docs/spec.md` 必須固定包含：

1. 目標與非目標  
2. 使用者流程 / operator 流程  
3. architecture 邊界  
4. failure model  
5. config/env contract  
6. 驗證方式  
7. rollout / rollback 邊界

**關鍵原則**：沒有 spec，不進 build。[^agent-skills]

#### C. `/plan`

**用途**：把 spec 分解成可平行、可驗證、可交付的 work packages。

**主要執行者**：

- `test-playbook`：先畫系統、先找風險，再決定測試層級。[^test-playbook]
- `coordinator`：把 task 切成 bounded jobs，避免 scope conflict。[^coordinator]

**建議文件責任分工**：

| 文件 | 角色 | 建議內容 |
| --- | --- | --- |
| `docs/spec.md` | canonical product/technical contract | what / why / boundaries |
| `docs/roadmap.md` | phase 序列 | phase 目標、依賴、進出條件 |
| `docs/test.md` | verification contract | unit / integration / e2e / stress / recovery / doc audits |
| `docs/task.md` | phase-level deliverables | 每 phase 的驗收任務 |
| `docs/todo.md` | active queue | 小粒度、可驗證、ready-to-run items |
| `docs/plan.md` | **當前 execution slice** | 本輪只做哪些 task、哪些 branch / worker 負責 |

這裡最重要的決定是：**`docs/plan.md` 不要再跟 `roadmap.md` 重複**。它應該只是一輪執行切片，而不是第二份 roadmap。

#### D. `/build`

**用途**：依 ready todos 平行開發。

**主要執行者**：

- `Max`：接收 build 指令與查詢狀態。[^max]
- `coordinator`：建立 job，控 scope、relay、replay。[^coordinator]
- `smux-manager-skill`：在 tmux panes 中做 manager/worker 協作。[^smuxmgr]

**建議做法**：

1. manager 只派發 **單一 slice** 給單一 worker
2. worker 任務 envelope 必含：
   - task id
   - 允許修改的 path
   - 驗證命令
   - artifacts 輸出位置
   - 上游 spec / task / todo 引用
3. 每個 worker 完成後只交：
   - diff / commit candidate
   - 驗證結果
   - 風險 / blocker
   - artifact path

這其實已經很接近 `autoresearch` 的 `program.md` 思路：讓 agent 在 bounded sandbox 裡工作，由人或 manager 去改「組織指令」，而不是讓 agent 任意擴大範圍。[^autoresearch]

#### E. `/verify`

**用途**：證明這個 slice 有效，不只是「看起來能動」。

你的 verify 最好拆三條線：

| 驗證線 | 工具 | 適用範圍 |
| --- | --- | --- |
| **程式驗證** | 專案原生 build/test + `test-playbook` | 一般 repo、library、service |
| **裝置/場域驗證** | `TestPilot` + `serialwrap` | prplOS / OpenWrt / UART / DUT workflow |
| **agent 工作流驗證** | `session-health` + `problemmap` | session 品質、收斂失敗、流程漂移 |

`TestPilot` 已經把 control plane 與 deterministic verdict kernel 分開；`serialwrap` 也已經提供單寫入仲裁與 UART 證據鏈。這兩者應該是你 hardware verify 的標準底座。[^testpilot][^serialwrap]

而對 agent workflow 本身，`session-health` 應該在每次大型任務後評估 SNR / STATE / CTX / REACT / DEPTH / CONV / TOOL；若 session-health 低於門檻，直接交給 `problemmap` 做 route-first 診斷。[^session-health][^problemmap]

#### F. `/review`

**用途**：在 merge / ship 前做質量與邊界審查。

建議拆兩種 review：

1. **code review**：找 bug、權限、配置漂移、驗證缺口
2. **doc review**：檢查 `spec / roadmap / test / task / todo` 是否與實作漂移

你現有流程裡最容易缺的其實不是 code review，而是 **doc drift review**。`test-playbook` 已經明確把 README / spec / usage drift 視為驗證表面；`agentrc` 也明確把 instruction drift / AI-readiness drift 放進 measure → generate → maintain cycle。[^test-playbook][^agentrc]

#### G. `/ship`

**用途**：讓「完成」真的變成可追溯的版本與部署事件。

**建議組合**：

- `conventional-commit`：生成 `zh-TW` Conventional Commit 訊息。[^convcommit]
- `picoclaw-ops-companion`：對 `git push`、部署、套件安裝、remote op 做 approval / audit。[^opscomp]
- `gitagent`：借用 agent repo 的 versioning / tag / branch promotion 思路。[^gitagent]
- `openclaw-obsidian-deploy` / `obs-service-handler`：沿用 `.env` + host mount + dynamic resolve 的部署習慣。[^openclaw-deploy][^obs-service-handler]

**ship 的最小合同**：

1. 有 commit  
2. 有 review 結果  
3. 有 verify artifact  
4. 有 version/tag 或至少 release note  
5. 有 deploy / rollback 記錄

## 4. 建議的角色分工

| 角色 | 核心責任 | 最適工具 |
| --- | --- | --- |
| **Manager** | intake、排程、指派、收斂、升級決策 | `Max`, `coordinator`, `smux-manager-skill` |
| **Researcher** | 盤點 repo / skill / external refs，建立 research doc | search tools, `graphify`, `obs-auto-moc query-memory` |
| **Planner** | 產 spec / roadmap / test / task / todo | doc workflow, `test-playbook` |
| **Builder** | 做 bounded implementation slice | repo native toolchain |
| **Tester** | 執行 program/device/session 驗證 | tests, `TestPilot`, `serialwrap`, `session-health` |
| **Debugger** | failure reduction、route-first triage | `problemmap`, logs, replay artifacts |
| **Reviewer** | code + doc drift review | review agent + doc checks |
| **Reporter** | 匯整 artifacts、更新 notes / dashboard / memory | markdown reports, `obs-auto-moc import-session-insights` |
| **Shipper** | commit/tag/push/deploy/rollback | `conventional-commit`, `picoclaw-ops-companion` |

### 關鍵規則

1. **只有 manager 可以改 task 狀態與派工**
2. **只有 shipper / guarded executor 可以執行高風險 push / deploy**
3. **reviewer 不直接改 code；builder 根據 review 回修**
4. **reporter 只寫 artifact / memory，不直接寫 canonical code**

這樣的 segregation 也和 `gitagent` 的 maker/checker/executor/auditor 分工思路一致。[^gitagent]

## 6. memory / 自主整理 / 自我修正的最佳落點

### 6.1 不要把 raw session 直接變成 canonical memory

你現在最好的做法，已經在 `obs-auto-moc` 裡落地：canonical scope 仍限 `root-note / TechVault / WorkVault / PersonalVault`，而 `memory/*` 是 secondary overlays；`import-session-insights` 也是吃 distilled `insights / lesson / decision`，不是 raw `.jsonl`。[^obsmoc-readme][^obsmoc-skill]

這一點非常重要，因為它讓你能同時保留：

- 可追溯的 notes truth
- agent operational learning
- dream / wake-up / persona 的 secondary synthesis

### 6.2 建議的 learning loop

#### 任務結束後（短週期）

1. `codex-lesson` 對單一 session 出 lesson artifact  
2. 若 session-health 分數太差，先跑 `problemmap`  
3. reporter 匯總成標準 artifact bundle  
4. `obs-auto-moc import-session-insights` 匯入 memory overlays  

[^codex-lesson][^session-health][^problemmap][^obsmoc-readme]

#### 每日 / 每輪（中週期）

1. `obs-auto-moc wake-up`
2. `obs-auto-moc dream`
3. `obs-auto-moc distill-persona-goals`

這讓你每天不是只看到 task list，而是看到：

- recurring frictions
- accepted working rules
- stale notes / contradictions
- persona heuristics

[^obsmoc-readme]

#### 每週 / 定期（長週期）

1. `codex-project-insights` 聚合多 session signals
2. 必要時 `evolve` 產出 AGENTS integration candidate
3. 但 **不要讓 evolve 自動變成 build hot-path**

`evolve` 適合當治理與規則維護層，不適合直接掛在每次 build 後自動改 AGENTS。它應該是週期性 maintenance gate。[^codex-insights][^evolve]

### 6.3 可以借但不需要全面導入的 memory pattern

| 來源 | 可借的點 | 不必照抄的點 |
| --- | --- | --- |
| `gbrain` | always-on signal detection、brain-first lookup、cron maintain、identity docs | 直接改用 PostgreSQL/pgvector 當核心記憶底座 |
| `mempalace` | local-first retrieval、agent diaries、knowledge graph、wake-up hooks | 把 verbatim conversation 全量變成主要 truth |
| `graphify` | graph cache、always-on architecture summary、queryable subgraph | 把所有工作流都建在 graph runtime 上 |

[^gbrain][^mempalace][^graphify]

## 7. config / env / 路徑治理原則

你已經多次表達「所有路徑都不要寫死」，而且你自己的 repo 也已經有正確示範：

- `obs-service-handler` 會從目前使用者的 `PATH`、`HOME`、`~/.config/obsidian-headless/sync/*/config.json` 自動解析 runtime 路徑，並允許 `OB_BIN` / `VAULT_PATH` 覆寫。[^obs-service-handler]
- `TestPilot` 對 `SERIALWRAP_BIN` 的解析順序是 `env → config → fail`。[^testpilot]
- `openclaw-obsidian-deploy` 也把 config 目錄與 vault 掛載外部化為 `.env` 參數，不把 auth / 個資塞進 repo。[^openclaw-deploy]

### 建議統一 contract

每個 workflow repo / meta repo 都遵守：

1. **CLI flag 優先**
2. **env 次之**
3. **config file 再其次**
4. **最後才是可推導 default**
5. **推不出來就 fail-close，不要猜**

### 建議標準變數

```bash
WORKFLOW_REPO_ROOT=
WORKFLOW_DOCS_ROOT=
WORKFLOW_RUNTIME_ROOT=
WORKFLOW_ARTIFACTS_ROOT=
WORKFLOW_MEMORY_ROOT=
WORKFLOW_REPORTS_ROOT=
WORKFLOW_TMUX_SESSION=
WORKFLOW_AGENT_LIST_PATH=
WORKFLOW_CONFIG_FILE=
```

如果未來做 meta repo，我建議再加一份：

- `config/workflow.yaml`
- `.env.example`
- `runtime/`（gitignored）
- `reports/`
- `docs/`

這樣就能跟 `gitagent`、`agentrc`、你自己的 deploy repo 保持一致風格。[^gitagent][^agentrc][^openclaw-deploy]

## 9. rollout 建議

### Phase 1：可落地 MVP

目標：把手動 prompt 串接改成「有規則的 manager-led flow」。

1. 固定 `/research` → `docs/research/<slug>.md`
2. 固定 `/define` → `docs/spec.md`
3. 固定 `/plan` → 產出 `roadmap/test/plan/task/todo`
4. 用 `coordinator + smux-manager-skill` 跑 `/build`
5. `/verify` 固定收：
   - repo tests
   - verify artifacts
   - `session-health`
6. `/review` 固定產 review report
7. `/ship` 綁 `conventional-commit` + guarded push
8. 將 `codex-lesson` / `codex-project-insights` 導入 `obs-auto-moc`

### Phase 2：把 workflow 從「可用」推到「可維運」

1. `Max` 成為預設入口
2. `session-health` 自動掛在大型 job 後
3. 失敗 session 自動轉 `problemmap`
4. `obs-auto-moc wake-up / dream` 變成每日節奏
5. `agentrc readiness/eval` 納入 CI

### Phase 3：把 workflow 從「可維運」推到「可擴張」

1. 導入 `graphify` 作為 codebase graph cache
2. 導入 `gitagent` 式 repo layout / duties contract
3. 讓 `Max` / `coordinator` 接 work board
4. 參考 `symphony` 把 work item 自動轉 isolated implementation runs

## 10. 不建議現在就做的事

1. **不要先做 full mesh**
   - 你現有工具天然偏 manager-led，不適合一開始就 agent 自由互講互改。

2. **不要先做一個新的 memory product**
   - `obs-auto-moc` 已有 note + ledger + wake-up + dream + persona + session import，先把它用滿。[^obsmoc-readme]

3. **不要讓 `evolve` 直接掛在每次 task 後自動改 AGENTS**
   - 它適合作為週期 maintenance gate，不適合 hot path。[^evolve]

4. **不要先造單一 web dashboard**
   - 先用 tmux + TUI + HTML report + Obsidian overlays 的 federated dashboard。

5. **不要再讓 docs 重複定義同一件事**
   - `spec`、`roadmap`、`plan`、`task`、`todo` 要有嚴格責任分工。

## 11. 最終建議

如果只給一個最精簡版本，我的推薦是：

> **用 `Max` 當入口、`coordinator + smux-manager-skill` 當 manager-led build runtime、`obs-auto-moc` 當長期記憶、`session-health + problemmap` 當 workflow 品質回饋、`picoclaw-ops-companion` 當 guarded ship/deploy executor。**

這條路的好處是：

- 幾乎全都建立在你已擁有、已理解、已部分落地的工具上
- 能先解決「手動 prompt 串步驟」問題
- 又不會因為過早做 full mesh / full autonomy 而失控
- 能自然接上你重視的 memory、dream mode、persona、dashboard、approval、agent 角色分工

---

## 12. 信心評估

### 高信心

- hub-and-spoke 比 full mesh 更適合你的現況
- `obs-auto-moc` 應該是 memory backbone，不該被新 memory system 取代
- `session-health + problemmap` 應該成為 workflow 自我修正閉環
- `picoclaw-ops-companion` 應該只承接高風險 action

### 中信心

- `Max` 是否最終應直接整合 `coordinator`，還是維持鬆耦合
- `graphify` 是否值得在所有 repo 全面部署，或只在大型 repo 啟用
- `agentrc` 要以 repo-by-repo 還是 meta-repo 方式導入

### 低信心 / 待後續驗證

- 是否真的需要 `symphony` 式 work-board 自動 intake
- 是否需要把 `gitagent` layout 全面套到你每個 repo
- `gbrain` / `mempalace` 是否在未來有必要補足 `obs-auto-moc` 缺的 retrieval 能力
