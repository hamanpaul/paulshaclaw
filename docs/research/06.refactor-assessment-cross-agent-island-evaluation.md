## 0. 定位（一句話）

> **paulshaclaw 的本體是「三家異質 CLI agent（Claude Code / Codex / Copilot）之上的 vendor-neutral 抽象層」——這層編排 + 記憶 + 治理，沒有任何單一 vendor、也沒有任何框架（LangGraph / CrewAI / AutoGen / ADK / MAF）提供。**

程式碼比重佐證：`memory/` 約 10,917 行（佔 package ~58%），其餘 monitor/core/cockpit/bot/lifecycle/persona/… 為相對薄殼。形狀＝「一顆記憶大腦 ＋ 一條 tmux/telegram 駕駛艙脊椎 ＋ 數個治理肢體」。

跨 agent 前提是**最關鍵的判準**：它翻轉了 v2 大半的「業界有更佳解」結論——因為那些現成解（Claude Code 原生 permissions、Claude memory…）幾乎都是**單 vendor**，只蓋三家中的一家。

## 2. v3 評估（跨 agent 前提）

### 2.1 翻轉摘要：哪些因「跨 agent」而從「業界更佳」變「必需且優於業界」

| 模組 | v2 判決 | **v3 判決（跨 agent）** | 翻轉原因 |
|---|---|---|---|
| `persona`/guardrail | 業界有更佳解（Claude 原生）| **必需且優於業界** | Claude 的 `permissions.deny`/hook 只管 Claude；Codex 的 approval+sandbox、Copilot 的 tool config 三家模型不同且不等強。**沒有現成的「跨 CLI agent RBAC」** |
| `security` AuditTrail | 必需（自製可接受）| **必需且優於業界** | 每家 vendor 只審計自己；**跨三家統一、防竄改的 audit** 無現成品 |
| `security` ApprovalGate | 部分重疊 Claude 權限 | **必需（放 coordinator 層）** | 「派工到任一 agent 之前」的 vendor-neutral 高風險攔截，單 vendor 權限管不到此位置 |
| `core` coordinator | 必需（脊椎）| **必需且優於業界** | 五大框架皆 in-process、單 provider API 編排；claude-squad/uzi 也只編排多個 Claude。**跨三家 OS-pane 異質編排，框架界無對應物** |
| workflow 選擇器（規劃）| LEAD 產品、MATCH 研究 | **超出研究前沿** | AdaptOrch/BAMAS 在單一模型生態內選 topology；**跨 vendor 同時選「隊形＋哪家 agent」無人做** |
| `memory` 引擎 + wakeup | LEAD | **LEAD 再強化** | 三家 session hook 正規化進同一庫、session-start 注入**跨家共享記憶**——Claude memory 是 Claude-only、Codex/Copilot 各自孤立 |
| `dream/proposals` | 接了優於業界 | 不變（優於業界）| proposal-first 本就 vendor-neutral、比 Mem0/A-MEM 的直接 mutate 更可審計 |

### 2.2 沒被「跨 agent」拯救的（仍是業界有更佳解）

關鍵區分：**這幾個管的是「系統自己的行程」，不是「三個 agent」**——跨 agent 前提套不上去，標準基礎設施仍贏。

| 模組 | 仍輸給 | 理由 |
|---|---|---|
| `observability` 的**服務監督** | systemd --user（`Restart=on-failure` / `MemoryMax=` / journald）| 監督 5 個自家 daemon 是 process 層、本就 vendor-neutral |
| `deploy/` | systemd 範本 unit `name@.service` + 30 行 install.sh | 部署與幾家 agent 無關；現為「規劃但不施工」的空殼 |
| `monitor` pub/sub | SQLite(WAL) + consumer poll | 跨 process 通訊不需為「三家 agent」特製 message bus |

> **observability 有一半被救回**：拆兩塊——「自家服務存活/重啟」→ systemd（業界贏）；「**三家 agent 的 liveness / 卡死 / 額度**」→ cross-agent health，systemd 不模型化，**此半邊自製正當**（額度部分 cost footer 已涵蓋）。

### 2.3 ⚠️ persona 翻轉後的硬約束（否則會做出「假安全」）

跨 agent 讓 persona 變必需，但**其保證強度 = 三家裡最弱那家的 enforcement**。三家原生強制力不對等（Claude Code 的 harness 層 PreToolUse hook 最強、Codex 有 approval+OS sandbox、Copilot 相對最弱）。正確接法為**雙層**：

1. **能 compile 就 compile 到原生**：persona contract → Claude `permissions.deny`/subagent `tools`、Codex config/sandbox、Copilot tool config。讓每家拿到**自己 harness 層**的強制（而非 prose 提示）。
2. **vendor 原生補不到的縫，在 coordinator/pane 層攔**（也是 ApprovalGate 的正位）。誠實標記：pane 層攔截比 harness 層弱、較易繞過。

→ persona 不該是「一套獨立 Python RBAC daemon」，而是「**一份 contract 編譯成 N 個 vendor backend ＋ 一層 coordinator 兜底**」。這個設計業界沒有、且是跨 agent 架構剛需。

## 3. ⭐ 重點業界對照：libtmux 與 OpenUsage

這兩個是研究中**最具體、最該立刻處置的「業界現成解」**——一個該被採用以刪自製碼，一個該被評估以決定是否停止維護自製碼。

### 3.1 libtmux —— cockpit 的底層管路該換掉，但獨特值要留

**現況（source）**：`cockpit/tmux.py:115-160` 等處**手解析 `tmux` 文字輸出**（`list-panes` / `capture-pane` / `swap-pane`）。

**業界對照**：
- 既有 tmux 工具分布清楚，**沒有任何一個做「跨 session pane 列表 + 判斷每個 pane 跑哪個 agent + 切 active」**：tmuxp/tmuxinator＝launcher、sesh＝session switcher、tmux-resurrect＝持久化、gotmux/libtmux＝控制庫。
- 原生 tmux 已給 primitive：`tmux list-panes -a -F '#{session_name} #{pane_pid} #{pane_current_command}'`、`choose-tree`、`select-pane`/`swap-pane`——但 `choose-tree` **沒有 agent 辨識**。
- **libtmux** 提供 typed `Session/Window/Pane` 走訪、`pane_current_command`、`select_pane()`。

**評估結論**：
- **cockpit 的「agent-aware dashboard」LEADS 其利基**——現成工具無法取代（差異化價值＝把 pane 對應到「哪個 AI agent」，`pane_current_command` 只會回 `node`/`python`/shell）。
- **但 (a) 列 pane 與 (c) swap 是 native tmux / libtmux 的薄包裝**。

> **處方**：cockpit 改坐在 **libtmux** 上（取代手解析 `tmux` 文字輸出），**只保留 agent 分類層 + Textual UI 當自製值**。這刪掉脆弱的字串切片（如 `start.sh:118-124` 風格的 tmux 輸出解析），降低維護面，且**跨 agent 前提下「agent 分類層」正是不可外包的核心**——三家 CLI agent 在 pane 裡長相不同，這個分類邏輯沒有現成品。

### 3.2 OpenUsage —— cost footer 唯一真正的 drop-in 候選，須先評估再決定是否續造

**現況（source）**：`cost/providers.py` 讀**真 quota 層**（非 token 估算）——Claude rate-limit sidecar、Codex `payload.rate_limits`、Copilot `/copilot_internal/user`，渲染 tmux `status-right` 三家額度同行。OOM 加固（64MB/mtime 邊界、兩層快取 + `O_EXCL` 單一 refresher）是全 repo 最實戰的工程。

**業界對照**：
- **不是重造 ccusage / claude-monitor**——那些是 **token 成本**、且 statusline **只有 Claude**。層級不同。
- **唯一真正對手 = OpenUsage**：已發布、積極開發、**34 provider**，用**完全相同的資料源**（`~/.claude`、Codex rate_limits、`/copilot_internal/user`），本地 HTTP API（`127.0.0.1:6736`），有 tmux 整合。

**評估結論**：
- cost footer **LEADS** vs ccusage/claude-monitor/claude-powerline/claude-code-statusline（這些 statusline 皆 Claude-only，或成本非配額）。
- **MATCHES OpenUsage**——這是唯一現實的 drop-in 替代。
- 在**跨 agent 前提下**，三家額度同行是核心需求；OpenUsage 正是衝著 cross-vendor 來的。

> **⚠️ 更正（2026-06-15 二次查證）**：本報告初稿曾依第一輪 README 表格判定「OpenUsage 的 tmux/statusline 僅支援 Claude」。二次直接查官方站與最新 repo 後，此判定**很可能有誤**——官方描述其 tmux 狀態列可「show your **Claude Code, Codex, Cursor, Copilot, and OpenRouter** usage（cost / quota / burn / active-tool）」，含 active-tool 偵測、品牌色 logo、`openusage tmux install`、12 套 preset。兩來源在此點衝突，**以實機驗證為準**。若屬實，則「三家同行 tmux 渲染」這個 footer 的差異化價值，OpenUsage 可能已具備。

> **補充事實（二次查證）**：
> - OpenUsage 以 **Go**（~95%）撰寫；footer 的 Python 採集邏輯**不可直接移植**，貢獻上游須以 Go 重寫。
> - 存在 **fork 歧義**：`janekbaraniewski/openusage`（原作）與 `robinebers/openusage`（另一 owner 的 copilot 文件）並存，下手前須確認 canonical upstream。
> - 對 Codex/Copilot，OpenUsage 文件用語為「rate limits / chat & completions quota」，但**未明確標示 `used_percent`/`percent_remaining` 欄位**——footer 需要的 quota% 是否齊備，須實測。

> **處方（先實測，再三選一）**：先跑 `openusage tmux install`，看三家實際輸出，據此決定——
> - **情況 A（最可能）：功能已存在** → footer 退役、改用 OpenUsage，省下全部自製採集碼維護。此時**不需要 PR**。
> - **情況 B：仍有真實缺口**（如 Codex/Copilot 只給 cost 非 quota%）→ 做 PR 才有意義；但須先開 issue 探 maintainer 意願與 `integrations` 外掛架構，並以 Go 重寫。
> - **情況 C：footer 真正獨到的是「採集穩健性」而非「多家渲染」** → 若 OpenUsage 的本地 log 掃描無邊界，則最該貢獻的 PR 是 **「OOM-bounded local-log scanning」**（64MB/mtime 邊界、`O_EXCL` 單一 refresher），這是跨所有 provider 受益、且你有 WSL 實戰證據的改進。
> - 一句話：**這是「續造 vs 採用 vs 貢獻」唯一需要實測才能拍板的子系統**——別在沒跑過 OpenUsage 之前就決定，也別在確認缺口前就動 PR。

## 4. 值得抽出來單獨成包（與接線狀態無關）

排序：
1. **`agent-eventlog`（memory ledger：append-only + hash-chain + decay/reactivation）**——文獻空白點（無人統一 append-only 事件日誌 ＋ 逐事件衰退/復活），最該開源。
2. **`deterministic-agent-memory`（importer + atomizer + moc + dream）**——「把 Karpathy LLM Wiki 做到工程級」，對外價值最大，且天生 vendor-neutral（三家 hook 已驗證）。
3. **`claude-gemma4-proxy`**（Anthropic→vLLM shim，零專案耦合）。
4. **`RedactionEngine` + `policy` 的 redaction/classification**（單檔 stdlib 小型 DLP）。

> §2.2 那組（observability 服務監督 / deploy / monitor-bus）**不值得抽包**——抽出來也輸給 systemd/SQLite，只是把重造的輪子包裝更精緻。

## 5. 簡化三刀（跨 agent 下不變，理由更強）

1. **`lifecycle` events 引擎 → 併進 `memory` ledger**：跨家 handoff 需要一本「誰在哪個 phase、artifact 走到哪」的共享帳；用既有 ledger 一本到底，別開第二套事件機制。
2. **一套 SQLite(WAL)** 收掉 ledger + `moc/search` + monitor 狀態 + 4 套 flock/fsync 與 relations O(n) 去重掃描。
3. **systemd --user + tmux（只給 cockpit）** 取代 313 行 bash supervisor（systemd 補上現缺的 restart 能力與資源上限）。

## 6. 評估演進註記（誠實揭露判斷如何被修正）

- **v1（中性）**：誤把「README 寫的 obs-auto-moc 待串接」當最大缺口 → 後經程式碼確認 MOC 已自建為 dream 第三 pass，README 過時。
- **v2（孤島皆 WIP）**：把「死島」改判為「接上後落在哪」；多數 vendor-neutral 治理層在「假設接上」後仍輸給 Claude Code 原生。
- **v3（再加跨 agent）**：因「需跨 Claude/Codex/Copilot 一致」，單 vendor 的現成解（Claude 原生 permissions/memory）不再是 drop-in 替代 → persona/audit/approval/coordinator/memory/選擇器**翻轉為「必需且業界沒有」**；僅「管自家行程」的 observability 服務監督 / deploy / monitor-bus 仍應外包給 systemd + SQLite。
