# `max/smux` 第一階段抽取研究：`tmux-bridge`、Telegram、Copilot SDK daemon、coordinator、pane-centric TUI

## Executive Summary

如果目標是**先做出可用的主核心與 TUI，而不是一次複製整個 `max` / `smux` 生態**，我建議的第一階段做法是：

1. **只抽 `tmux-bridge`，不要整包搬 `smux`**。你短期真正需要的是 pane bus，不是整套 tmux UX opinion。`tmux-bridge` 已經具備 read-before-act guard、socket 自動偵測、pane label、跨 pane message header、pane list / read / type / keys / doctor，這些正好就是 agent 溝通底層。[^tmux-bridge-read-guard][^tmux-bridge-socket][^tmux-bridge-list][^tmux-bridge-message][^tmux-bridge-read][^tmux-bridge-doctor]
2. **控制核心應該做成「PaulShiaBro daemon」**：保留 `max` 的 daemon + persistent orchestrator session + local API/SSE 這條骨架，但不要直接沿用它的 chat-first TUI。你的需求是 pane/task/operator-first，不是終端聊天室。[^max-daemon][^max-api][^max-tui][^max-orchestrator]
3. **短期 task manager 不要重寫，直接吸收你現有 `coordinator` 的狀態模型**。它已經有 provider defaults、worker modes、read/write scope conflict、job/mailbox/log/relay 結構，足夠當多 agent 調度基礎。[^coord-core][^coord-runtime][^coord-conflict][^coord-relay]
4. **Telegram 先維持 token + authorized user 模式**。`max` 現況也是這樣，而且官方並沒有提供「BotFather API 自動建 bot」這種正式 provisioning API；若要做長期自動化，應設計成「半自動註冊流程」或另外的 operator bootstrap，而不是假設有現成 API。[^max-setup][^max-env][^max-telegram][^telegram-botfather]
5. **從現在就把 transport 抽象化**。第一階段 backend 用 `tmux-bridge`，但 registry / agent identity / API contract 不要把 pane ID 寫死。中長期若改成 UNIX socket / ACP server / MCP server，才不需要整套重做。[^tmux-bridge-list][^sdk-options][^sdk-session]

一句話總結：**短期做 `tmux-bridge + coordinator + PaulShiaBro daemon + pane TUI`，中長期再把 transport backend 從 tmux pane 換成 socket / ACP / MCP。**

## 我驗證到的現成 building blocks

### 1. `tmux-bridge` 已經足夠當短期 agent bus

本機 `tmux-bridge` 不是黑盒 binary，而是可讀 shell script。它目前已經提供：

- read-before-act guard
- `TMUX_BRIDGE_SOCKET -> $TMUX -> scan socket -> default tmux server` 的 socket fallback
- `list/read/type/keys/message/name/resolve/id/doctor`
- `message` 自動帶 sender pane / session-window 資訊
- `list` 輸出 pane id、session/window、size、actual process、label、cwd

這些能力直接覆蓋你短期需要的：

- 看所有 pane
- 給 pane 命名
- 把訊息投給別的 agent pane
- 擷取 pane snapshot
- 做基本 diagnostics

也就是說，**短期不需要先做自己的 tmux RPC daemon**；把 `tmux-bridge` 包成 transport adapter 就夠。[^tmux-bridge-read-guard][^tmux-bridge-socket][^tmux-bridge-list][^tmux-bridge-message][^tmux-bridge-read][^tmux-bridge-doctor]

### 2. `max` 上游其實很接近你要抽的「最小核心」

上游 [`max`](https://github.com/burkeholland/max) 的骨架非常清楚：

- `daemon.ts`：SQLite / wiki / Copilot SDK / orchestrator / HTTP API / Telegram bot 啟動鏈
- `api/server.ts`：local loopback + bearer token + SSE stream
- `telegram/bot.ts`：單一 authorized user、text / photo handling、proactive notify
- `copilot/orchestrator.ts`：persistent orchestrator session、message queue、model routing、background completion 回灌
- `copilot/tools.ts`：worker session 管理、machine session attach、skills / memory facade
- `tui/index.ts`：chat-first CLI TUI，透過 local API + SSE 跟 daemon 互動

這意味著你不需要從零設計「always-on Copilot SDK service」：**可以直接把 `max` 當成 reference architecture，再把不符合你需求的 UI 與 product features 剪掉。**[^max-daemon][^max-api][^max-telegram][^max-tools-workers][^max-orchestrator][^max-tui]

### 3. 你本地的 `coordinator` 已經有 task manager 雛形

本地 `coordinator.py` 已經有幾個非常關鍵的東西：

- provider defaults：`codex / copilot / gemini`
- worker modes：`auto / none / template / sdk`
- job / mailbox / log / relay 狀態目錄
- `read_scope` / `write_scope` 與 overlap conflict block
- `sdk` mode 走 `copilot_sdk_orchestrator.mjs`
- 非 `copilot` provider 透過 `sdk_delegate_template` 委派

這些比單純「起幾個 pane 然後互傳 prompt」成熟很多，因此**第一階段 task manager 不該重寫，而應直接把 pane registry 與 `coordinator` job model 接上。**[^coord-core][^coord-runtime][^coord-conflict][^coord-relay]

### 4. Copilot SDK 有用，但邊界要看清楚

Copilot SDK 確實支援你要的幾個核心能力：

- `cliPath` / `cwd` / `useStdio` / `cliUrl`
- `workingDirectory`
- `tools`
- `mcpServers`
- `customAgents`
- `skillDirectories`
- `infiniteSessions`

但也有兩個很重要的限制：

1. `cliUrl` 和 `cliPath/useStdio` **互斥**；若你接的是外部 server，auth 與 process lifecycle 就不再由目前這個 client 管。[^sdk-options][^sdk-runtime]
2. `autoRestart` 在目前 SDK type 裡是 **deprecated / no effect**；如果你想做穩定 daemon，不要把恢復能力寄望在 SDK option 上，而要自己做 service supervision / restart policy。[^sdk-options][^sdk-runtime]

這代表你中長期若要做「用 exec path 啟動 codex / gemini / Claude Code」這種多 provider controller，**真正要抽象的是 provider runtime layer，不是硬 patch 在單一 Copilot session 概念裡。**[^sdk-options][^sdk-session][^sdk-runtime]

## 我建議的第一階段架構

```text
                         Human / Operator
                              │
          ┌───────────────────┼───────────────────┐
          │                   │                   │
          ▼                   ▼                   ▼
     Telegram            Pane TUI            CLI / hooks
          │                   │                   │
          └───────────────────┬───────────────────┘
                              ▼
                    PaulShiaBro control daemon
             (persistent orchestrator + local API)
                              │
       ┌──────────────────────┼──────────────────────┐
       │                      │                      │
       ▼                      ▼                      ▼
  agent registry         task registry         event / notify bus
 (agent_id↔endpoint)     (job/mailbox/log)     (SSE / relay / alert)
       │                      │
       ▼                      ▼
 transport adapter       provider runtime adapter
 tmux-bridge now         copilot sdk / delegate cli
 unix socket later       codex / gemini / claude later
       │
       ▼
 tmux panes / pane snapshots / swap / send / capture
```

核心原則是：**控制平面、任務平面、傳輸平面分離。**

### 1. Control plane：做一個 `PaulShiaBro daemon`

第一階段最值得保留的是 `max` 的 daemon 形狀，而不是整個產品功能：

- 啟動時建立 state dir、DB、registry
- 建一個 persistent orchestrator session
- 提供本機 API 給 TUI / Telegram / hooks
- 做 background completion notifications
- 做 cancel / restart / health surface

這條鏈在 `max` 裡已經是成熟形狀：daemon 先起 DB / wiki，再起 Copilot client，再建 orchestrator session，再掛 proactive notify，然後開 HTTP API 與 Telegram bot。[^max-daemon]  
對你的版本，我建議把 wiki / remember / recall 這些產品層能力先降到最低，保留：

- daemon lifecycle
- orchestrator lifecycle
- local API
- notification routing
- worker / job visibility

### 2. Transport plane：只先做 `tmux-bridge` backend

你短期提的需求其實都能映射到現成 tmux primitives：

| 需求              | 短期做法                                                 | 直接依賴                 |
| --------------- | ---------------------------------------------------- | -------------------- |
| 顯示所有 pane       | `tmux-bridge list` 或 `tmux list-panes -a`            | `tmux-bridge` / tmux |
| 顯示 pane 對應任務    | registry 補欄位，不靠 tmux 原生                              | 你自己的 state           |
| 顯示 pane 快照      | `tmux capture-pane -p -J -S -N` / `tmux-bridge read` | tmux                 |
| 傳訊息到 agent pane | `tmux-bridge message + keys Enter`                   | `tmux-bridge`        |
| 手動控制 pane       | `type / keys / read`                                 | `tmux-bridge`        |
| 交換 pane 位置      | `tmux swap-pane -s %X -t %Y`                         | tmux                 |
| 聚焦 / 選 pane     | `tmux select-pane -t`                                | tmux                 |
| 標記 pane 名稱      | `tmux-bridge name` / `set-option -p @name`           | `tmux-bridge` / tmux |

你查到的 tmux 原生命令也證明這條路是穩的：`capture-pane`、`list-panes`、`select-pane`、`split-window`、`swap-pane` 都是現成能力；tmux 甚至已內建 prefix 綁定的 `swap-pane -U/-D` 與 pane menu。[^tmux-commands][^tmux-keys]

因此第一階段不要急著做 socket transport；**只要把 transport interface 設計對**：

- `listEndpoints()`
- `readSnapshot(endpoint, lines)`
- `sendText(endpoint, text)`
- `sendKeys(endpoint, keys[])`
- `labelEndpoint(endpoint, label)`
- `health()`

現在 backend 實作成 `tmux-bridge`；之後再新增 `unix-socket` backend 即可。

### 3. Registry plane：不要把 pane ID 當永久 agent identity

這是你目前最容易漏掉的地方之一。  
如果直接把 `%7`、`%11` 這種 pane ID 當 agent identity，短期會很方便，但後面會很痛。建議至少拆兩層：

- `agent_id`：穩定身份，例如 `builder-A`、`reviewer-2`
- `endpoint_id`：目前掛在哪裡，例如 `pane:%7`、未來可能是 `sock:/tmp/agent-builder-A.sock`

Registry 至少要存：

| 欄位 | 用途 |
| --- | --- |
| `agent_id` | 穩定身份 |
| `role` | builder / tester / reviewer / manager |
| `provider` | copilot / codex / gemini / claude |
| `model` | `gpt-5-mini` 等 |
| `reasoning` | `high/xhigh` 等 |
| `cwd` | 工作目錄 |
| `transport` | `tmux-pane` / `unix-socket` |
| `endpoint` | `%7` / socket path |
| `job_id` | 目前任務 |
| `status` | idle / busy / blocked / dead |
| `last_seen_at` | 心跳 |
| `owner` | 誰能寫入這個 agent |

這樣 TUI 才能顯示你要的：

> `%7 [copilot gpt-5-mini high] builder-A — research serialwrap`

### 4. Task plane：短期直接接 `coordinator`

`coordinator` 已經有 job / mailbox / relay / scope conflict。第一階段最合理的做法不是再發明一個 task manager，而是：

1. **`coordinator` 當 source of truth**
   - `job_id`
   - `trace_id`
   - `from_agent / to_agents`
   - `read_scope / write_scope`
   - `mailbox / logs`
2. **daemon 補一層 agent registry**
   - 哪個 agent 目前掛哪個 pane
   - 哪個 job 對到哪個 pane / provider
3. **TUI 只做 operator console**
   - 看 job
   - 看 pane
   - 看 mapping
   - 發送 control action

短期 provider strategy 也已經很清楚：

- `copilot`：走 `sdk` mode
- `codex/gemini`：先走 `sdk_delegate_template` 或 `template` mode

等第二階段再把這些 provider runtime 收斂成同一個 exec-path adapter。[^coord-core][^coord-runtime][^coord-conflict]

### 5. Telegram：先做 adapter，不做 provisioning platform

上游 `max` 的 Telegram 其實很適合你短期抽：

- 僅允許單一 `AUTHORIZED_USER_ID`
- text / photo 都能轉成 prompt + attachments
- 可推 background 完成通知
- 可從本機 daemon 主動送照片與訊息

這些都符合你「手機上看結果、遠端 dispatch 任務」的需求。[^max-telegram]

但有兩個現實要先講清楚：

1. `max setup` 現在明確就是**手動引導你去找 `@BotFather` 建 bot、去找 `@userinfobot` 查 user ID，再回填 `.env`**。[^max-setup][^max-env]
2. Telegram 官方文件目前仍要求透過 `@BotFather` 互動建立 bot，沒有正式公開的「BotFather 建 bot API」。[^telegram-botfather]

所以你的中長期目標若寫成「直接接 BotFather API 自動初始化部署」，建議改寫成：

> **設計 operator-assisted bootstrap / semi-automated provisioning flow**

而不是假設能走正式公開 API。

### 6. TUI：不要沿用 chat-first，而要做 pane-first

上游 `max tui` 很成熟，但它本質上是：

- 連本機 HTTP API
- 用 `/stream` SSE 接回覆
- 支援 slash commands
- Escape cancel

這很適合聊天，不完全適合你要的「pane dashboard」。[^max-tui][^max-api]

我建議你的 TUI 第一版改成 3-pane operator console：

1. **左欄：agents / panes list**
   - label
   - pane id
   - provider / model
   - status
   - current job
2. **中欄：snapshot preview**
   - 選定 pane 的最近 N 行
   - ANSI sanitize 後顯示
3. **右欄：job / mailbox / relay**
   - job status
   - trace id
   - pending asks / reviews / answers

底部 actions：

- focus pane
- swap pane
- send message
- interrupt
- open cwd
- replay relay

也就是說，**第一版 TUI 不該先做「Max 的另一個聊天殼」，而應該做 tmux operator dashboard。**

## 你目前還沒完全納進來、但實際很重要的面向

### 1. Agent identity 與 endpoint identity 要分開

這是第一個一定要先補的設計點。  
如果不分，未來一旦 pane 重啟、pane 被 swap、session attach/detach，整個 job mapping 會亂掉。

### 2. Concurrent writes / exclusive ownership

你已經有 `coordinator` 的 `write_scope` overlap block，但 pane transport 本身還沒有「誰可以對這個 pane 發命令」的 ownership 邊界。  
如果 reviewer 和 manager 同時往同一個 pane 送字，會直接把 prompt 弄髒。

建議至少加：

- `exclusive_controller`
- `pending_input_lock`
- `last_read_at`
- `last_write_at`

而且要把 `tmux-bridge` 的 read guard 視為 transport safety，不是 workflow safety。[^tmux-bridge-read-guard]

### 3. Snapshot sanitize

pane preview 不是純文字；它常帶 ANSI、progress bar、partial redraw、敏感路徑甚至 token-like 字串。  
如果你的 TUI 要做 `<prefix> s` 類似 preview，必須先決定：

- 是否保留 ANSI 顏色
- 如何去掉控制碼
- 多行重繪如何摺疊
- 是否要遮蔽 credential-like 字串

否則 pane preview 很容易從「方便」變成「難讀 + 洩漏資訊」。

### 4. Cancel / restart / recovery contract

`max` 自己已有：

- queue serialization
- cancel current message
- recoverable error retry
- session resume / recreate
- daemon restart flow

但你若把 pane agents、Telegram、TUI、coordinator 全接進來，還要再定義：

- job cancelled 之後 pane 要不要保留
- pane 死掉後 job 狀態怎麼補
- attach 到既有 session 後是否允許 takeover
- daemon 重啟後 registry 如何重建

這塊若不先定義，系統會很快進入「看得到 pane，但不知道它算不算活著」的狀態。[^max-orchestrator][^max-daemon][^max-tools-workers]

### 5. Backpressure 與 worker memory cost

`max` 在 worker 完成後會 auto-destroy session，還直接在註解寫出大約每個 worker 會吃掉大量記憶體，因此不能無限制留著。[^max-tools-workers]

這提醒你兩件事：

1. **長駐 orchestrator + 短命 workers** 是比較合理的。
2. pane-based agents 如果也長駐，TUI 必須清楚區分：
   - persistent operator panes
   - ephemeral task panes

不然記憶體和上下文都會越積越亂。

### 6. Provider abstraction 不能綁在 Copilot session 名詞上

上游 `max` 的 workers 本質仍然是 `CopilotSession`；你未來想支援 `codex/gemini/Claude Code` 時，不能直接把所有 runtime 都硬塞進這個 model。[^max-tools-workers][^sdk-session]

建議先抽一個 provider runtime contract：

- `spawn()`
- `send()`
- `cancel()`
- `snapshot()`
- `destroy()`
- `capabilities()`

然後：

- `copilot-sdk` runtime 實作一版
- `cli-delegate` runtime 實作一版

這樣第二階段才不會全系統都寫死在 Copilot 概念上。

### 7. Security boundary 要分三層，不只 token

你現在想到的安全可能還比較偏 token / bot token；但這套系統至少有三層：

1. **control plane auth**
   - 本機 API bearer token
   - Telegram authorized user
2. **transport auth**
   - tmux pane visibility
   - 未來 socket file permission / owner / group
3. **task scope auth**
   - read_scope / write_scope
   - 哪個角色可操作哪種 agent

其中第三層如果不做，前兩層再安全也只能保護「外人不要進來」，保護不了「內部 agent 不要越界」。

### 8. `autoRestart` 不能當 daemon strategy

這點特別重要。  
如果你想把主控 agent 做成 service / daemon，**請直接把 restart 放在 systemd / supervisor / 你的 own daemon lifecycle**，不要以為 SDK `autoRestart` 會幫你。當前 SDK 明確把它標成 deprecated / no effect。[^sdk-options][^sdk-runtime]

### 9. 非 tmux agent 也要能掛進同一 registry

你中長期明確想走 socket / ACP / MCP。那代表從第一版開始，registry 就應該容納：

- tmux pane agent
- detached daemon worker
- remote MCP-backed worker
- attached existing Copilot session

上游 `max` 已經示範了「attach machine session」這種概念，這很好，但也提醒你：**transport 與 identity 要從第一版就分離。**[^max-tools-workers]

## 建議的分階段落地順序

### Phase 1A — 最小可用控制核心

目標：

- 起一個 `PaulShiaBro daemon`
- 建 persistent orchestrator session（`gpt-5-mini` 為主）
- 建 agent registry + task registry
- transport 只接 `tmux-bridge`
- 只做 read-only pane TUI

完成標誌：

- 可以列 pane
- 可以看 pane snapshot
- 可以看到 pane↔agent↔job 對照
- 可以看 coordinator job / mailbox / relay

### Phase 1B — 可操作 pane fleet

目標：

- send message / send keys
- select / swap pane
- 從 TUI dispatch 任務到 `coordinator`
- 用 registry 維護 ephemeral / persistent panes

完成標誌：

- TUI 可對指定 agent 發 prompt
- 可以從 job 反查 pane，從 pane 反查 job
- `write_scope` conflict 會阻擋不安全 dispatch

### Phase 1C — Telegram 遠端操作面

目標：

- 接入 Telegram text / photo
- background completion notify
- 限縮成單一 authorized user

完成標誌：

- 手機可下任務
- 任務完成可收到通知
- 可從手機拿到 pane / job 摘要

### Phase 2 — Runtime abstraction 與 socket transport

目標：

- provider runtime interface
- `cli-delegate` runtime
- UNIX socket transport backend
- 將 `pane:%7` 升級為 `endpoint` 抽象

完成標誌：

- 同一個 TUI 可同時看 pane agents 與 socket agents
- 不需要 tmux 也能啟動 agent

### Phase 3 — ACP / MCP / 更完整服務化

目標：

- ACP / MCP server surfaces
- 更細緻的 policy / audit / review gate
- cross-host / remote orchestration

這一階段才適合談：

- 更完整的 agent mesh
- server-backed attachment
- formalized review / governance services

## 最終建議

如果只給一個最實際的方案，我會建議你：

> **第一階段不要做「Max clone」或「smux clone」，而是做一個以 `tmux-bridge` 為 transport、以 `coordinator` 為 task state、以 Copilot SDK 為主控 session 的 `PaulShiaBro operator core`。**

具體上：

- 抽 `tmux-bridge`
- 參考 `max` 保留 daemon / API / Telegram / orchestrator
- 吸收 `coordinator` 當 job / mailbox / relay / scope 核心
- 重寫一個 pane-first TUI
- 從第一天就把 `agent_id` 與 `endpoint_id` 分離

而你目前最該立即修正的設計假設有三個：

1. **不要把 pane ID 當永久 agent identity**
2. **不要假設有官方 BotFather provisioning API**
3. **不要依賴 SDK `autoRestart` 做 service recovery**

只要先把這三個點校正，第一階段就能做得很穩，第二階段也比較容易演進到 socket / ACP / MCP。

- Copilot SDK 可以當 controller，但不應承擔整個多 provider runtime abstraction

### Medium confidence

- 將 `max` 的某些 session / worker lifecycle 直接移植到你的系統時，仍需依你實際 token / memory / provider cost policy 做裁剪
- UNIX socket / ACP / MCP 的第二階段細節，還要再看你最終想支援的是 local-only 還是 cross-host

### Low confidence / explicit caveat

- Telegram bot provisioning 的全自動化目前不應當成近期可落地目標；官方公開能力看起來仍不支援直接 API 建 bot，只能設計 operator-assisted bootstrap。[^telegram-botfather]
