# Stage 4 研究：persona — 角色目錄、handoff、guardrail、記憶 loadout

## Executive Summary

Stage 4 要解決的核心問題是：**「誰有資格執行什麼」這件事，目前還是靠 prompt 與 operator 默契在維持。** `01.prompt-...md` §4 已經列過 9 個角色，但沒定義：他們的能力邊界、可用 tool、可寫路徑、可載入的記憶子集、用哪個 provider / model、失敗時該 escalate 給誰。Stage 4 就是把這些寫成**可被 gate engine 判讀的 persona contract**。

三個最重要的判斷：

1. **persona ≠ agent instance ≠ skill**。persona 是「角色契約」（能力 + 義務 + 邊界），agent instance 是「目前跑這個角色的那條 session」，skill 是「角色執行任務時會用到的工具」。混在一起會讓 guardrail 失效。
2. **persona 應該綁 phase，不是綁 provider**。Stage 3 的每個 phase 有允許 persona 清單；provider / model 是 persona 的屬性之一，隨時可換（cost 考量、降級、A/B）。
3. **User 本人也是一個 persona**。從 `obs-auto-moc distill-persona-goals` 產出的 heuristics / goals 是一份長期守則，lifecycle runtime 要把它當成所有 persona 共享的底層 instruction，而不是只給 manager 看。

基於此，Stage 4 最適合的形態是：

- **Persona registry**：一份版本化的 `personas.yaml`，列出所有角色的 contract
- **Persona loader**：agent 啟動時根據 `AGENT_ROLE` env 載入對應 instruction + memory scope + tool allowlist
- **Handoff protocol**：persona 間的溝通走 coordinator relay，不走 pane direct write
- **Guardrail engine**：讀 persona contract，對每個 tool call / file write 做 pre-check
- **Persona-aware gate**：Stage 3 gate engine 認 persona；phase entry gate 含「由授權 persona 提交」

## 1. 為什麼 persona 是獨立 stage，不是 Stage 1 的附加項

### 1.1 Stage 1 只夠承擔 endpoint / transport / task dispatch

Stage 1 的 agent registry 記的是 `agent_id`、`endpoint`、`provider`、`model`、`job_id`。它不知道：

- 這個 agent **應該**做什麼（能力契約）
- 這個 agent **允許**做什麼（權限契約）
- 這個 agent 失敗時**誰接手**（escalation 契約）
- 這個 agent 該讀**哪些記憶**才能工作（context 契約）

這四件事全部屬於 persona 層。如果硬塞進 Stage 1，registry 會臃腫到無法治理。

### 1.2 Stage 2 只夠承擔記憶儲存 / 分類 / 索引

Stage 2 不知道「Reviewer 不能讀 experience/hotfix-*.md 以外的 incident 記錄」、「Builder 只載當前 slice 的 plan 與相關 code 摘要」這類規則。這些是 persona → memory 的 scope mapping，需要獨立一層。

### 1.3 Stage 3 定義 phase 邊界，但不定義角色邊界

Stage 3 的 gate 關心「artifact 有沒有」「schema 對不對」。它不關心「是誰產出」。沒有 Stage 4，同一個 agent 可以今天當 reviewer 明天當 shipper，guardrail 就失效。

因此 Stage 4 的存在理由：**把「身份」從 endpoint 與 memory 裡抽出來，變成獨立契約**。

## 2. 核心抽象：persona = contract、agent = instance、skill = capability

### 2.1 三層關係

```
User / Operator
      │
      ▼
Persona (contract, versioned, in personas.yaml)
      │  "Reviewer v1.2: 可讀全部 code、不可寫 code、可寫 reports/review/*"
      ▼
Agent instance (runtime, in Stage 1 registry)
      │  "agent-reviewer-01 掛在 pane %7, provider copilot, model gpt-5"
      ▼
Skill (tool, in custom-skills/)
      │  "test-playbook, problemmap, doc-drift-scan"
```

### 2.2 三層各自的生命週期

| 層 | 生命週期 | 誰定義 | 誰變更 |
|---|---|---|---|
| Persona | 長，版本化 | User + evolve | PR + 審查 |
| Agent instance | 短，會銷毀 | Manager / lifecycle runtime | 任務結束即回收 |
| Skill | 中，獨立演進 | Skill 作者 | 版本號管理 |

### 2.3 為什麼這樣切

- Persona 長期穩定 → 可以用於教學、審計、drift check
- Agent instance 短命 → 符合 max 的 worker auto-destroy 思路
- Skill 獨立演進 → 不同 persona 可共用 skill；skill 變動不會強迫所有 persona 改版

### 2.4 Persona vs Skill：兩個正交維度，不是互斥

常見誤解：「如果我已經有 `review-skill`，讓 agentA 去 review agentB 的結果就好了，為什麼還要 persona？」

**答**：這個例子本身就**同時用到了兩層**。只是 persona 層通常被隱形吸收進 "agentA" 這個代稱裡。展開來看：

| 維度 | Skill | Persona |
|---|---|---|
| 回答什麼 | **做什麼**（procedure） | **誰在做 / 以什麼身份做**（actor） |
| 是否帶授權？ | 否，中性 procedure | 是，帶 scope + guardrail |
| 是否帶記憶偏好？ | 否 | 是，decide memory loadout |
| 是否綁失敗升級？ | 否 | 是，escalation chain |
| 可重用性 | 跨 persona 共用 | 跨 slice 穩定 |
| 範例 | `review-skill`、`test-playbook`、`problemmap` | `reviewer`、`builder`、`shipper` |

**你說的「agentA 以 review-skill review agentB 的結果」拆開來是：**

- `agentA` = persona instance（例如 `reviewer`）
  - 決定**授權**：能不能擋 merge、能不能寫哪個路徑
  - 決定**記憶載入**：載 `reports/review/*` 歷史、載當前 slice 的 plan.md
  - 決定**失敗升級**：blocked 時把 handoff 交給 `operator` 還是 `manager`
- `review-skill` = 中性 procedure
  - 定義 review 的**步驟**（讀 plan → 讀 diff → 套 rubric → 產出 report）
  - 不知道呼叫者是誰，也不管呼叫者有沒有授權

**只有 skill 沒有 persona 會失去：**

- 授權控管：任何 agent 都能呼叫 `ship-skill`、把主線 push 出去
- Memory loadout：每個 agent 看到的 context 一樣，`reviewer` 拿不到 review history、`builder` 拿不到當前 diff
- 升級路徑：blocked / 超出 scope 時沒人接手
- 審計：事件流只看得到「skill 被呼叫」，看不到「哪個角色呼叫的」

**只有 persona 沒有 skill 會失去：**

- 流程可重用性：每個 persona 都要在 instruction file 裡寫一遍「怎麼做 review」，同一個流程被複製 N 份，drift 難以控制
- Skill 版本治理：skill 更新時沒有單一升版入口，persona instruction 各自承載版本

**結論**：Stage 4 的 persona contract **不取代 skill**，而是替 skill 提供 caller 身份 + scope + memory context。persona 綁 phase（Stage 3 決定哪個 phase 可以誰進）、persona 呼叫 skill（Skill 是 persona 手上的工具），兩者各司其職。

## 3. 角色目錄（persona catalog）

### 3.1 九個核心 persona

沿用 `01.prompt-...md` §4 的角色列表，但補上契約層。

| Persona | 核心責任 | 可寫 | 可讀 | 主要 tool / skill | 建議 provider |
|---|---|---|---|---|---|
| Manager | intake / 排程 / 派工 / 升級 | `lifecycle.yaml`、`docs/plan.md`（當前 slice 欄位） | 全 | coordinator, smux-manager-skill | copilot gpt-5 |
| Researcher | 盤點、產 research doc | `docs/research/<slug>.md` | 全 docs + 外部 | obsidian-cli, graphify, web | codex / gemini（看成本） |
| Planner | spec → roadmap/test/task/todo | `docs/spec.md`, `roadmap.md`, `test.md`, `task.md`, `todo.md` | 全 docs | test-playbook, agentrc | copilot gpt-5 |
| Builder | bounded implementation slice | `task.allowed_paths` 宣告的路徑 | 全 code + current plan | repo-native toolchain | copilot gpt-5-mini（多實例） |
| Tester | 執行 program / device / session 驗證 | `reports/verify/*` | 全 code（read-only） | test runner, TestPilot, serialwrap, session-health | codex（便宜跑多線） |
| Debugger | failure reduction、triage | `reports/debug/*`, `experience/incidents/*` | 全 code + logs | problemmap, logs, replay | codex / copilot |
| Reviewer | code + doc drift review | `reports/review/*` | 全（只讀） | review agent, doc-drift-scan | copilot gpt-5（品質優先） |
| Reporter | 匯整 artifacts、更新 notes / memory | `~/.agents/memory/inbox/*`, `reports/summary/*` | 全（只讀） | obs-auto-moc import, codex-lesson | gemini / claude haiku（便宜） |
| Shipper | commit / tag / push / deploy / rollback | `reports/ship/*`, git tags, remote | 全（只讀） | conventional-commit, picoclaw-ops-companion | copilot gpt-5（需判斷力） |

### 3.2 兩個額外 persona

除了 `01` 列出的九個，Stage 4 建議再加：

| Persona | 角色 | 特點 |
|---|---|---|
| Operator（**人類**）| 人在迴路的決策者 | 能跑所有 slash command；有 `/rewind`、`gate override`、`ship --skip-approval` 權 |
| Memory Janitor | dream service 的執行者 | 只寫 `runtime/dream/*`、`proposals/*`；不動 canonical |

Operator 明確建模成 persona 的好處：所有事件流仍可歸類、審計，人類行為不會變成 blind spot。Memory Janitor 獨立成 persona 也能保證 dream service 不會越界改 canonical（對應 Stage 2 proposal-first 原則）。

### 3.3 Persona 命名規則

- 角色名小寫 kebab-case：`manager`, `researcher`, ..., `memory-janitor`
- Instance 加序號：`agent-builder-01`, `agent-builder-02`（多 builder 並行時）
- Persona 版本獨立：`reviewer@1.2`，用於 rollback 或 A/B

## 4. Persona contract schema

### 4.1 contract 欄位

```yaml
personas:
  reviewer:
    version: 1.2.0
    summary: "對 slice 內 code 與 docs 做最終審查；不直接修改產物"
    allowed_phases: [review]
    disallowed_phases: [build, ship]
    scope:
      read:
        - "**/*"
      write:
        - "reports/review/**"
        - "runtime/lifecycle/events.jsonl"   # 只能 append 自己角色的事件
    tools:
      - review-agent
      - doc-drift-scan
      - test-playbook
      - problemmap
    denied_tools:
      - git commit
      - git push
      - gh pr merge
    memory_loadout:
      common_sense: [coding-style, review-checklist, incident-patterns]
      project_scope:
        - spec.md
        - plan.md (current slice)
        - test.md
        - reports/verify/<current-run>
      knowledge_subset:
        tags: [review, drift, quality]
    handoff:
      on_approve: ship
      on_changes_requested: build
      on_block: manager
    escalation:
      blocked_by_scope_conflict: manager
      blocked_by_ambiguous_spec: planner
      blocked_by_failing_gate: manager
    provider_binding:
      primary: {provider: copilot, model: gpt-5, reasoning: high}
      fallback: {provider: codex, model: gpt-5-mini, reasoning: xhigh}
      cost_cap_per_run_usd: 1.00
    drift_signals:
      - never_cite_rule: "review report 必須引用 spec section"
      - max_avg_latency_min: 30
      - min_findings_per_run: 0   # 不強制找 bug，但要求明確 verdict
    instruction_files:
      - personas/reviewer/base.md
      - personas/reviewer/zh-TW.md   # 使用者語言 overlay
    evolution:
      owned_by: user
      last_updated: 2026-04-19
      last_lesson_ref: "codex-lesson://2026-04-15-review-missed-hw-case"
```

### 4.2 為什麼 contract 要這麼重

- **gate engine 要能自動判讀**：沒有 schema 就無法自動 guardrail
- **evolve 要能精準更新單一欄位**：`evolve` skill 產出的 AGENTS 建議，對應到 persona 的哪個欄位要有明確座標
- **多 provider 測試要能做 A/B**：`provider_binding` 單獨成欄位，利於跑 shadow run

### 4.3 contract 的版本與遷移

- `version` 走 semver：major 表示 handoff / scope 改動，minor 表示 tool / instruction 改動，patch 表示文字修正
- 舊 slice 不強制升版；Stage 3 gate engine 以 slice 記錄的 persona version 為準

## 5. 個別 persona 深入設計

### 5.1 Manager

**特殊性**：唯一能跨 phase 的 persona；唯一能下 `/rewind`、`escalate` 的非人角色。

**風險**：若 Manager 過度主動改 artifact，會把 hub-and-spoke 退化成單人 agent。**Manager contract 必須明確禁止修改 non-管理類 artifact**。

**建議**：

- Manager 只寫 `lifecycle.yaml`、`plan.md` 的「slice 分派欄位」、`runtime/lifecycle/events.jsonl`
- Manager 不直接產 spec / code / test / review
- 遇到 gate 失敗，Manager 召集對應 persona 重跑，不自己補

### 5.2 Researcher

**特殊性**：最需要**外部讀取**的角色（web、obsidian、graphify）。

**風險**：外部資料污染；未核實引用。

**建議**：

- 強制 frontmatter `citations` 至少 N 條；每條必須含 URL 或本地 path
- Researcher 不寫 `docs/spec.md`；只寫 `docs/research/<slug>.md`
- 使用 `obs-auto-moc query-memory` 讀既有筆記，但不寫 canonical

### 5.3 Planner

**特殊性**：產出五份文件，承擔跨文件 cross-ref 正確性。

**風險**：spec/roadmap/test/task/todo 五份文件內部不一致。

**建議**：

- 強制 planner 輸出時跑 cross-ref 檢查：每個 `task.id` 必須有對應 `roadmap.phase`、每個 `todo.id` 必須有對應 `task.id`、每個 `test.case` 必須有對應 `spec.section`
- 失敗當 gate 失敗處理，不當 rework

### 5.4 Builder（多實例）

**特殊性**：唯一預設可以有多個並行 instance 的 persona。

**風險**：兩個 builder 同時改同一檔案；scope 重疊。

**建議**：

- 每個 builder instance 綁一個 todo；todo 的 `allowed_paths` 就是 builder 的 write scope
- 多 builder 必須通過 coordinator 的 write_scope conflict check
- 多 builder 失敗時 Manager 仲裁合併順序

### 5.5 Tester / Debugger

**拆開的理由**：Tester 執行既定測試、Debugger 做 route-first triage；兩者 mind-set 不同。Tester 期望 deterministic，Debugger 期望 exploratory。

**建議**：

- Tester 的 `drift_signals` 含「verify 耗時 vs 基準」
- Debugger 只在 gate 失敗或 ship 後事件觸發；不常駐
- 兩者共用 `reports/` 但寫不同子目錄

### 5.6 Reviewer

**特殊性**：絕對不能改 code 或 ship artifact。

**建議**：

- `denied_tools` 顯式列出 `git commit`、`git push`、`gh pr merge`
- verdict 枚舉 `approve / changes_requested / block`
- `block` 時必須 escalate 到 Manager，不能自動 loop 回 Builder

### 5.7 Reporter

**特殊性**：最常被低估；但它是 Stage 2 記憶品質的把門人。

**建議**：

- Reporter 不直接寫 `work-centric/`；只寫 `inbox/`，讓 Stage 2 importer 處理
- Reporter 的 `instruction_files` 要綁住 frontmatter provenance schema（Stage 2 §3.3）
- Reporter 跑在便宜 model（gemini / claude haiku），因為量大

### 5.8 Shipper

**特殊性**：唯一能觸發不可逆動作的 persona。

**建議**：

- 所有 Shipper action 都必須經 `picoclaw-ops-companion` approval
- Shipper 有獨立的 `cost_cap_per_run_usd`；shipper 跑得少但每次值得用貴 model
- Shipper 失敗（push 被拒、tag 衝突）不自動重試；必寫入 `experience/ship-failures/*`

### 5.9 Operator（人類）

**特殊性**：有最高權限，但仍要留痕。

**建議**：

- Operator 的所有 slash command 都寫進 `events.jsonl`，actor = `operator:<telegram-id>`
- Operator 可 override gate，但 override 必帶 `--reason`
- Operator 可下 `/rewind`；非 Operator persona 不行
- Operator 的遠端入口（Telegram bot）必須 **24x7 常駐**，否則 Operator 無法下指令；但這份 always-on 屬於 Stage 1 infrastructure（PaulShiaBro daemon 的 Telegram listener），**不是 Operator persona 本身的屬性** — Operator 作為人類，活性由真人決定

### 5.10 Memory Janitor

**特殊性**：需要 **24x7 背景常駐**的 persona；但它不是系統中唯一常駐的元件。

> 系統中有三個獨立的 always-on 成員，職責不同、失敗域分離：
> 1. **PaulShiaBro daemon**（Stage 1 infra）— 含 Manager orchestrator session、local API、事件匯流
> 2. **Telegram bot listener**（Stage 1 infra）— Operator 的遠端入口
> 3. **Memory Janitor service**（本 persona）— 24x7 dream / wake-up / synthesis 背景整理
>
> 三者必須能**獨立重啟**；任何一個掛掉不影響其他兩個（詳見 Stage 5 / overview 的獨立性原則）。

**建議**：

- 只寫 `runtime/dream/`、`proposals/`
- 對 canonical 改動必走 proposal-first + Operator 批准
- drift_signals 含「N 天無 proposal 產出」「queue backlog 過高」
- 自身 crash 時 Stage 1 daemon 不受影響；重啟由 Stage 5 supervisor 觸發

## 6. Provider / model binding

### 6.1 選擇 binding 的三個維度

- **品質**：review / ship / manager 用貴 model
- **成本**：builder / reporter 可用便宜 model（並行多實例）
- **可用性**：fallback provider 必備（copilot 當掉時 codex/gemini 接手）

### 6.2 預設 policy（建議起點）

| Persona | Primary | Fallback | 備註 |
|---|---|---|---|
| Manager | copilot / gpt-5 | codex / gpt-5 | 需判斷力 |
| Researcher | codex / gpt-5-mini | gemini | 量大讀外部 |
| Planner | copilot / gpt-5 | codex / gpt-5 | 結構化強 |
| Builder | copilot / gpt-5-mini | codex / gpt-5-mini | 多實例 |
| Tester | codex / gpt-5-mini | gemini | 量大 |
| Debugger | copilot / gpt-5 | codex / gpt-5 | 需判斷 |
| Reviewer | copilot / gpt-5 | codex / gpt-5 | 品質優先 |
| Reporter | gemini / flash | claude / haiku | 量大且便宜 |
| Shipper | copilot / gpt-5 | — | 不 fallback，失敗要 operator 接手 |
| Memory Janitor | claude / haiku | gemini / flash | 長期運轉 |

### 6.3 Binding 不綁死 runtime

- binding 寫在 `personas.yaml`，可 hot reload
- Stage 1 agent registry 啟動 instance 時讀 binding；已啟動 instance 不受影響
- 實驗時可用 `--persona-override reviewer.primary=claude/sonnet` 做 shadow run

### 6.4 成本控管

- 每個 persona 有 `cost_cap_per_run_usd` 與 `cost_cap_daily_usd`
- 超過 daily cap 時該 persona 暫停；Manager 升級通知 Operator
- 成本記錄寫 `runtime/lifecycle/cost.jsonl`

## 7. Persona memory loadout

### 7.1 三塊 loadout

| 塊 | 來源 | 用途 |
|---|---|---|
| Common sense | `work-centric/common-sense/` 子集 | 每個 persona 看不同 subset；Reviewer 看 review-checklist、Builder 看 coding-style |
| Project scope | `work-centric/<project>/` 依 phase 子集 | Builder 看 plan + code、Reviewer 看 spec + verify |
| Knowledge subset | `knowledge/` 依 tag 檢索 | 由 persona contract 的 `knowledge_subset.tags` 指定 |

### 7.2 載入策略

- Agent instance 啟動時，loader 依 contract 組 context bundle
- 大量內容：reference by path，不 inline（避免 context 爆）
- 長期 user heuristics：inline（短；必讀）
- 專案當前 slice 的 `plan.md` / `spec.md`：inline
- 歷史 experience / knowledge：reference + lazy fetch

### 7.3 Loadout 版本

- context bundle 有 checksum，寫進 agent instance metadata
- gate engine 可驗「這次執行載的是哪個版本的 spec」

## 8. Persona instruction file

### 8.1 三層 instruction

| 層 | 範圍 | 誰擁有 |
|---|---|---|
| `personas/_shared/base.md` | 所有 persona 共用（禮節、安全、語言偏好） | User + evolve |
| `personas/<role>/base.md` | 單一 persona 行為規範 | User + evolve |
| `personas/<role>/<locale>.md` | 語言 overlay（zh-TW 等） | User |

### 8.2 與 AGENTS.md / CLAUDE.md / GEMINI.md 的關係

- Stage 4 不直接用 AGENTS.md 做 persona carrier
- Stage 4 的 instruction 透過 wrapper 注入 env 或 prompt prefix
- AGENTS.md 仍存在（repo-level policy），但內容應 reduce 到「指向 `~/.agents/personas/`」，不重複寫

### 8.3 zh-TW overlay 的必要性

User heuristics（從 `obs-auto-moc distill-persona-goals`）含大量 zh-TW 偏好。不做語言 overlay 的話，英語系 base prompt 會吃掉這些偏好。建議：

- base.md 保持 en（跨 provider 穩定）
- `zh-TW.md` overlay：語言、文化脈絡、命名偏好、報告格式
- loader 依 `LANG` / `AGENT_LOCALE` 決定是否載 overlay

## 9. Handoff / escalation 協議

### 9.1 Handoff 的四種形態

| 類型 | 觸發 | 走哪 |
|---|---|---|
| 正常序列 | phase pass | lifecycle runtime 依 Stage 3 gate 自動派下一 persona |
| 條件 branch | verdict = changes_requested | Reviewer → Builder（帶 findings） |
| Escalation | blocked / scope conflict | 任何 persona → Manager |
| Fail-open | gate override | Operator → 任意 persona |

### 9.2 Handoff message schema

```yaml
handoff:
  from: reviewer@1.2
  to: builder
  reason: changes_requested
  slice: slice-2026-04-20-a
  findings:
    - path: src/x.py:42
      severity: block
      note: "error path 未覆蓋"
  artifact_refs:
    - reports/review/2026-04-20T11.md
  expires_at: 2026-04-21T00:00Z
```

走 coordinator relay；不直接 pane-to-pane。

### 9.3 Escalation chain

```
Any persona ─┐
             ├─ blocked by scope conflict → Manager
             ├─ blocked by ambiguous spec → Planner
             ├─ blocked by failing gate   → Manager
             ├─ cost cap exceeded         → Operator
             └─ unknown                   → Manager → Operator
```

每條 escalation 必寫 `events.jsonl`。

### 9.4 Expires_at 的重要性

handoff 若超過 `expires_at` 未接，自動升級到 Manager。避免 builder 無聲卡住。

### 9.5 Slice roster 與 co-worker awareness

> **問題**：§9.1–9.4 定義的是「點對點」handoff，但每個 persona 在**跑任務當下**並不知道：此 slice 上還活著哪些 peer（例如 builder-02 正在改哪個檔、reviewer 正在盯哪份 diff、tester 正在跑哪批 case）。缺乏這份「誰在旁邊」的感知，會引發三種混亂：
>
> 1. **跨 slice 誤接**：builder 把屬於 slice-A 的 findings 當成 slice-B 的修
> 2. **同 slice 撞檔**：builder-01 / builder-02 同時改 `src/x.py`
> 3. **幽靈 handoff**：reviewer 送 changes_requested 給已死的 builder instance

#### 9.5.1 `slice_roster`：slice 層的花名冊

每個開啟中的 slice 有一份活花名冊，由 Manager 維護、coordinator 廣播、lifecycle runtime 保存：

- 路徑：`runtime/lifecycle/slices/<slice_id>/roster.yaml`
- 寫入者：**只有** coordinator（經由 `roster.*` event）；任何 persona 都**只讀**
- 版本：每次變更 bump `rev:`，供 loader 判斷是否需要 re-sync

```yaml
slice_id: slice-2026-04-20-a
phase: build
rev: 7
updated_at: 2026-04-20T11:42:00Z
members:
  - persona: manager
    version: 1.1
    agent_id: agent-manager-main
    pane: "%3"
    provider: {copilot, gpt-5}
    status: active        # active | idle | blocked | finished | quarantined
    joined_at: 2026-04-20T09:00:00Z
    owns:
      artifacts: ["lifecycle.yaml", "plan.md#slice-2026-04-20-a"]
      claims: []
  - persona: builder
    version: 1.0
    agent_id: agent-builder-01
    pane: "%7"
    provider: {copilot, gpt-5-mini}
    status: active
    joined_at: 2026-04-20T10:12:00Z
    owns:
      artifacts: ["reports/build/slice-2026-04-20-a/01.md"]
      claims: ["src/feature/x.py", "src/feature/x_util.py"]   # 路徑鎖
  - persona: builder
    version: 1.0
    agent_id: agent-builder-02
    pane: "%8"
    provider: {copilot, gpt-5-mini}
    status: blocked
    blocked_reason: "waiting on reviewer@1.2 for 01.md"
    joined_at: 2026-04-20T10:15:00Z
    owns:
      artifacts: ["reports/build/slice-2026-04-20-a/02.md"]
      claims: ["src/feature/y.py"]
  - persona: reviewer
    version: 1.2
    agent_id: agent-reviewer-01
    pane: "%9"
    provider: {copilot, gpt-5}
    status: active
    joined_at: 2026-04-20T11:30:00Z
    owns:
      artifacts: ["reports/review/slice-2026-04-20-a/01.md"]
      claims: []
handoffs_open:
  - from: builder@1.0(agent-builder-02)
    to: reviewer@1.2
    reason: ready_for_review
    expires_at: 2026-04-20T13:00:00Z
gate_state:
  last_phase_passed: plan
  current_phase: build
  next_phase: review
```

欄位語義：

| 欄 | 作用 |
|---|---|
| `members[].claims` | 路徑鎖。guardrail §10.1 檔案系統層寫入前會查此表；**同 slice 內禁止兩個 agent 宣告重疊 claim** |
| `members[].status` | 驅動 TUI 色與 handoff 受理邏輯；`finished / quarantined` 的 agent 不可成為 handoff `to` 目標 |
| `members[].owns.artifacts` | 供 reviewer / reporter 回查「這份 report 是誰產的」而不需要翻 frontmatter |
| `handoffs_open` | §9.2 handoff 的 live view；超過 `expires_at` 自動升級至 Manager（§9.4） |
| `rev` | loader 單調遞增序號；agent re-sync 比對 `rev` 決定是否重讀 |

#### 9.5.2 Event 流：roster 如何被更新

roster 不是輪詢，是**事件驅動**；所有變更寫 `events.jsonl` 後由 coordinator 投影回 `roster.yaml`：

| 事件 | 誰發 | 後果 |
|---|---|---|
| `roster.joined` | persona loader（agent 啟動完成） | append member、bump rev |
| `roster.left` | lifecycle runtime（agent 正常結束 / 回收） | mark `finished`、釋放 claims |
| `roster.status_changed` | persona 自報（active ↔ idle ↔ blocked） | 更新 status |
| `roster.claim_added` / `roster.claim_released` | guardrail 層 | 更新 `claims`；衝突則拒絕並回 error |
| `roster.handoff_opened` / `roster.handoff_closed` | coordinator relay（§9.2） | 同步 `handoffs_open` |
| `roster.quarantined` | guardrail（§10.3 多次違規） | status=quarantined，不再受理 handoff |

#### 9.5.3 Persona 側：awareness 如何進到 agent 腦裡

配合 `05.paulshaclaw-...md` Stage 2 驗收的 pull-based 原則（**不依賴 push**），roster 資訊走兩條路進 agent context：

1. **Bind-time 注入**（bootstrap）
   - persona loader 在 `memory_loadout` 之後、系統 prompt 合成之前，讀當前 slice 的 `roster.yaml`
   - 濾掉自身，投影成一段 `<co_workers>` block 塞進 system prompt：

     ```
     <co_workers slice="slice-2026-04-20-a" rev="7">
     - manager@1.1 (agent-manager-main, pane %3, active) owns plan.md, lifecycle.yaml
     - builder@1.0 (agent-builder-01, pane %7, active) claims src/feature/x.py, src/feature/x_util.py
     - builder@1.0 (agent-builder-02, pane %8, blocked: waiting on review) claims src/feature/y.py
     open handoffs: builder-02 -> reviewer (ready_for_review, expires 13:00Z)
     </co_workers>
     ```

2. **Task-envelope 注入**（pre-phase hook，§Stage 2 l.406）
   - lifecycle runtime 派每個 task 時，把 roster 的 `rev` 與 peer 摘要塞進 task envelope 的 `context.peers`
   - persona 在收到 task 時優先讀此欄位；若 envelope `rev` < loader 快取 `rev`，loader 觸發 re-sync（SIGHUP，§Stage 2 l.408）

3. **On-demand 查詢**
   - skill `roster-inspect`（開放給所有 persona）：`roster-inspect --slice <id>` 回當前 rev 的 roster 摘要
   - reviewer 常用於「這份 report 01.md 的作者 agent 現在還活著嗎？」

#### 9.5.4 三種混亂的對應機制

| 混亂 | 防線 | 位置 |
|---|---|---|
| 跨 slice 誤接 | 每個 task envelope 帶 `slice_id`；persona guardrail 檢查 envelope `slice_id` 是否 == 自己 bind 時的 slice，不符拒絕並發 `roster.mismatch` 事件 | §10 guardrail + §9.5.2 |
| 同 slice 撞檔 | `claims` 路徑鎖；guardrail §10.1 寫入前查 roster，命中他人 claim 拒絕；衝突寫 `roster.claim_conflict` 升 Manager | §10.1 + §9.5.1 |
| 幽靈 handoff | handoff `to` 前 coordinator 先查 roster 該 persona 是否 `active`；非 active → 嘗試備援實例（同 persona 其他 agent_id）或升級 Manager | §9.2 + §9.5.2 |

#### 9.5.5 與 Stage 1 / 3 / 5 的具體介面

- **Stage 1**：§13.1 的 TUI pane 標籤擴為 `%7 [copilot gpt-5-mini] builder@1.0#01 — slice-0420a [active] claims:2`
- **Stage 3**：phase entry gate 多一線「roster 中 current_phase 的 `allowed_phases` 匹配者至少一人 `status=active`」；`/rewind` 會觸發 `roster.reset` 把舊 phase 的 claims 全釋放
- **Stage 5**：dashboard 直接投影 roster.yaml；`experience/<slice>.md` 收尾時把 roster 最終狀態拍扁成一欄「參與者時間軸」

#### 9.5.6 驗收：slice roster / co-worker awareness

> 以下為可測、可 CI 驗的 checkpoint。對應 `05.paulshaclaw-...md` Stage 4 驗收清單擴充項。

- [ ] **R1 Roster 存在性**：任一 slice 進入 `build` phase 後 5 秒內，`runtime/lifecycle/slices/<slice_id>/roster.yaml` 存在且 `members[]` 非空
- [ ] **R2 事件一致性**：每次 `roster.*` 事件在 `events.jsonl` 出現後，`roster.yaml` 的 `rev` 嚴格遞增 1；差值 ≠ 1 觸發 health incident
- [ ] **R3 Bind-time 注入**：隨機抽樣 agent instance，dump 其 system prompt，必含 `<co_workers slice=... rev=...>` block；block 中 `rev` 必等於 bind 當下 roster.yaml 的 `rev`
- [ ] **R4 Task envelope 注入**：任一 task envelope 的 `context.peers` 為該 slice roster 的投影（除自身外）；缺欄位則 lifecycle runtime 拒派
- [ ] **R5 Slice 隔離**：人工注入一張帶錯誤 `slice_id` 的 task envelope，guardrail 必拒絕並寫 `roster.mismatch` 事件；該 agent 本身不受影響
- [ ] **R6 Claim 互斥**：同 slice 兩個 builder 申請重疊路徑 claim，第二者必被拒；`roster.claim_conflict` 事件存在且升級到 Manager
- [ ] **R7 幽靈 handoff 防護**：手動把某 builder 設為 `finished`，reviewer 送 `changes_requested` 至該 persona，coordinator 必改派至同 persona 其他 `active` agent 或升 Manager，**不**靜默送往死 agent
- [ ] **R8 Re-sync**：長跑 agent（>15 min）在 roster `rev` 前進時，15 秒內收到 SIGHUP 並 dump 更新後的 `<co_workers>`；`memory_state` 不回 degraded
- [ ] **R9 On-demand skill**：所有正式 persona 均可呼叫 `roster-inspect`；denied_tools 中**不得**列此 skill
- [ ] **R10 Pane 標籤**：TUI 顯示之 pane 標籤含 `persona@version#instance` 與 `slice_id`；與 roster.yaml 對應 member 一致
- [ ] **R11 Slice 收尾**：slice 關閉時 `experience/<slice>.md` 必含「參與者時間軸」欄位（persona × joined/left/blocked timestamps），由最終 roster 投影產生
- [ ] **R12 Roster 降級**：coordinator 掛時，persona loader 仍以 bootstrap manifest 上次成功快取的 roster 啟動，標 `roster_state: degraded`；降級期間禁止新 claim，僅允許只讀與 escalation

驗收執行：golden slice 回歸測試（`05` Stage 3 l.426）加一條「roster 完整生命週期」子場景，含啟動兩個 builder、一個 reviewer、注入撞檔、注入幽靈 handoff、注入 coordinator crash，涵蓋 R1–R12。

## 10. Guardrail engine

### 10.1 Guardrail 的四層

1. **檔案系統層**：wrapper 對 write syscall 對照 `scope.write` 檢查
2. **工具層**：tool invocation 對照 `tools` / `denied_tools`
3. **phase 層**：phase dispatch 對照 `allowed_phases`
4. **成本層**：token / cost 達 cap 時拒絕新 call

### 10.2 實作建議

- 檔案系統層用 Stage 1 wrapper（launch agent 前注入 env + `LD_PRELOAD` 或 `fakeroot`-style guard）
- 工具層用 skill 宿主包裝：skill 註冊時帶 persona filter
- phase 層走 lifecycle runtime 的 dispatch 階段
- 成本層由 coordinator 收 token usage 統計

### 10.3 Guardrail 失敗的處理

- 預設 fail-close：違規 call 拒絕 + 寫 event
- 不靜默 fallback（違規寫嘗試不自動改寫到允許路徑，會 hide bug）
- 多次違規觸發 drift：該 persona 進入 `quarantined`，Operator 審核才解除

## 11. User persona overlay（你自己）

### 11.1 為什麼 User 也要建模成 persona

- 所有 persona 都要讀「User 的長期偏好」；這份偏好若只存在於 Manager prompt，其他 persona 會漂移
- Stage 2 的 `obs-auto-moc distill-persona-goals` 已經在蒸餾這份偏好；Stage 4 要把它變成可載入的 context

### 11.2 User persona 的內容

- 語言偏好（zh-TW）
- 工作風格（hub-and-spoke、artifact-first、不重造輪子）
- 禁忌（不喜歡過度 abstraction、不接受 silent fallback）
- 判斷啟發（偏好 pointer 而不是 verbose、偏好事件流而不是 RPC）
- 命名 / branding（破蝦哥 / PoHsiaBro）

### 11.3 與其他 persona 的合成

- 每個 persona 的 base.md 由 `personas/_shared/user-overlay.md` 做 include
- overlay 版本有 checksum；Stage 3 gate engine 可驗所有 agent 是否跑同版
- 當 overlay 版本改時，Manager 必須通知 User 確認，不自動擴散（避免 evolve skill 誤覆蓋）

### 11.4 Drift 的來源

- codex-lesson 產出新 heuristic 時，是新增到 User overlay 還是 persona base？原則：
  - 跨 persona 普遍適用 → User overlay
  - 單一 persona 專屬 → persona base
  - 一次性現象 → experience/，不進 overlay

## 12. Persona evolution 與 drift detection

### 12.1 Evolution loop（借 evolve skill）

- 每次 slice 關閉後，Reporter 把 session 透過 `codex-lesson` 蒸餾
- `evolve` skill 聚合多份 lesson，提案對特定 persona contract 的修改
- 提案寫到 `runtime/persona-proposals/<role>/<date>.md`
- Operator review 後 merge 到 `personas.yaml`

### 12.2 Drift detection

每個 persona 定義 `drift_signals`，週期性檢查：

- 行為 drift：某 persona 連續 N 次跳過必做步驟（如 reviewer 連續 5 次沒引用 spec）
- 成本 drift：平均 cost 超過基準 X%
- 品質 drift：reviewer verdict distribution 偏離歷史
- 延遲 drift：平均 run time 超過閾值

Drift 事件不自動改 contract，寫進 `runtime/drift/<role>/<date>.md`；Operator review。

### 12.3 Persona A/B 測試

- 用 `version` 同時跑兩版：`reviewer@1.2`（production）和 `reviewer@1.3-rc`（shadow）
- shadow 不影響 gate，但產出同樣寫入 `reports/review/`，前綴 `shadow-`
- 比對完後 Operator 決定 promote

## 14. 治理盲點

1. **Persona 粒度**：9+2 是起點，日後可能需要 Architect、Ops、Security 等。加新 persona 的審核流程沒定。
2. **單人雙角色**：使用者一人兼 Operator + 技術判斷者。escalation 到 Operator 時若睡覺怎麼辦？需要 `operator.after_hours.policy`。
3. **Persona 私有記憶**：某些 persona 會累積「只給自己看」的筆記（debug 私人 notes）。要不要做 per-persona private memory？建議**不做**；統一走 knowledge + tag。
4. **多專案不同 persona 需求**：某些專案需要「保守版 Reviewer」，某些需要「寬鬆版」。建議用 persona version + project binding，不另建 persona。
5. **Persona 離線失效**：如果 provider 全掛，persona fallback chain 應如何降級？需要「離線 operator-only 模式」的明確契約。
6. **新手 User 匯入**：未來 onboarding 新 user 時，User overlay 從空白開始，早期沒有 heuristic。需要 seed heuristic 集。
7. **Persona 對 skill 版本的依賴**：skill 升 major 可能破壞 persona。需要 skill 版本 pin 在 persona contract。

## 16. 不建議現在就做

1. **不要做 web-based persona editor**。YAML + git PR 已足夠；UI 會讓 drift 更難追。
2. **不要讓 evolve 自動 merge persona 改動**。proposal 必須人工核准。
3. **不要給每個 persona 配 private memory**。會污染 Stage 2 的 clean 分層。
4. **不要一開始就 10+ persona**。從 3 個起步驗證 handoff；再擴。
5. **不要把 persona 和 skill 混在同一個 registry**。skill 可共用，persona 是身份；混了之後無法分 scope。

## 17. 結論

Stage 4 的價值不在於「角色變多」，而在於把**「誰有資格做什麼」做成 diffable / auditable / evolvable 的檔案**。三條紅線：

- **persona 與 agent instance 與 skill 嚴格分層**
- **contract 可被 gate engine 判讀，不只給人看**
- **User 本人明確建模成 persona**

只要這三條到位，Stage 3 的 lifecycle gate、Stage 2 的記憶分類、Stage 1 的 agent registry 都能穩定地互相引用，整套系統才真正長成 hub-and-spoke 而不是「看起來像 hub 但其實是 mesh」。

## 18. Confidence assessment

### 高信心

- persona ≠ agent instance ≠ skill 的三層分法
- Reviewer / Shipper 必須 denied_tools 明確禁寫 code 與 push
- User 建模成 persona 是長期最重要的一步
- guardrail 要 fail-close，不要 silent fallback

### 中信心

- provider binding 的預設 mapping（會隨 cost / availability 變）
- persona 數量（9+2 是否過多或過少）
- drift_signals 的閾值

### 低信心 / 待驗證

- 多 builder 並行時的 coordination 細節（依賴 coordinator 的 write_scope 成熟度）
- persona A/B shadow run 是否值得（成本高、收益未知）
- persona private memory 是否真的不需要（可能在某些 debug 場景會被打臉）

## 19. 附錄 A：最小 personas.yaml 範例

```yaml

shared:
  user_overlay: personas/_shared/user-overlay.md
  base: personas/_shared/base.md

personas:
  manager:
    version: 1.0.0
    summary: "intake / 派工 / 升級；不產 spec / code / review"
    allowed_phases: [research, define, plan, build, verify, review, ship]

      read: ["**/*"]

        - "docs/plan.md"
        - "runtime/lifecycle/events.jsonl"
    tools: [coordinator, smux-manager-skill, problemmap]
    denied_tools: [git commit, git push]
    provider_binding:
      primary: {provider: copilot, model: gpt-5, reasoning: high}
      fallback: {provider: codex, model: gpt-5, reasoning: high}
```
