## 1. 專案本質

**paulshaclaw 是一人份的 agent 工作流編排系統**。

它不是：

- 多租戶平台
- 產品化 CLI
- GBrain / MemPalace 替代品
- Max / smux 的完整 clone

它**就是**：

- 讓 Operator（使用者自己）能在 TUI + Telegram + CLI 上對一群帶角色的 agent 下指令
- 讓這些指令走**可追溯、可 gate、可 replay** 的 lifecycle
- 讓所有產出整理成**分層記憶**供未來的 agent 載入
- 讓這一切在本機常駐、失敗可修、升級可控

## 3. 核心設計原則（所有 stage 共用）

1. **Hub-and-spoke**：Manager 是唯一 task authority，worker 只做 bounded execution。不做 full mesh。
2. **Artifact-first + event-first**：任何階段沒有 artifact 落地 + 事件寫入，就等於沒發生過。Prompt 不是真相來源。
3. **Proposal-first**：對 canonical 記憶的任何改動先走提案流程，不直接回寫（dream service、evolve 都是 proposer）。
4. **Fail-close**：guardrail、security gate、memory write 遇到異常**拒絕**，不自動降級放行。
5. **Stage 獨立性**：下游 stage 失敗不應擊倒上游（詳見 §7）。
6. **借觀念不造輪子**：優先使用已有資產（custom-skills、obs-auto-moc、max、smux、session-health、TestPilot、serialwrap、problemmap），缺什麼才加。
7. **路徑不寫死**：CLI flag → env → config → fail-close；`XXX_ROOT` 類變數可覆寫（延續 obs-service-handler / TestPilot / openclaw-deploy 慣例）。

## 4. 三軸分層架構（部署 layout）

### 4.1 三類資產、三類路徑

| 類別 | 路徑 | 是否進 repo | 包含 |
|---|---|---|---|
| **Code / Template** | `paulshaclaw/` repo | 公開 | daemon 程式、TUI、skill 擴充、所有 state/memory 的 `.sample` / `.template` 檔、banner 素材、font 建置腳本、文件 |
| **State / Memory** | `~/.agents/` | 獨立 **private** repo | registry、lifecycle events、inbox、work-centric、knowledge、runtime、health；其中 `~/.agents/memory` → **symlink** → `~/notes/paulshaclaw/memory`（obs vault 下獨立資料夾，不混入 obs 本體筆記） |
| **Secret / 機敏** | `~/.config/paulshaclaw/` | **絕不進 repo** | Telegram bot token、Copilot / API tokens、authorized user id、私鑰、帶實機 secret 的 `.env` |

### 4.2 部署契約（Stage 7 草案）

- Core repo 的 template 檔以 `.sample` / `.template` 後綴命名；`psc install` 第一次執行時：若目標路徑不存在 → copy + rename；若存在 → skip 並記 log
- `~/.agents/` 的 private repo 有自己的 commit 歷史，用於個人備份 / 同步；不與 `paulshaclaw/` 合併
- `~/.config/paulshaclaw/` 有 `*.env.example` 在 `paulshaclaw/config/` 作為 template，但本體只存本機

### 4.3 Memory symlink 佈局

```
~/notes/paulshaclaw/memory/            ← 實體資料（obs vault 底下獨立資料夾）
    ├── inbox/
    ├── work-centric/
    │   ├── common-sense/
    │   └── <project>/
    ├── knowledge/
    └── runtime/

~/.agents/memory → ~/notes/paulshaclaw/memory   (symlink)
~/.agents/
    ├── config/
    ├── memory -> ...                  ← 上面的 symlink
    └── state/                         ← 非 memory 的 runtime state
```

此佈局取代了原 Stage 9 的「既有資產匯入」mission（該 mission 已永久取消，由本節 symlink 接手）。**Stage 9 編號於 2026-04-26 重新啟用**，新 mission 為 Project Monitor service（見 §5.1）。本節 symlink 的好處：**obs vault 原有筆記完全不動**、新記憶資料享有 obs sync / backup / preview、`~/.agents/memory` 語意統一給所有 agent。

### 5.2 Stage 0 的必要性（為何不從 Stage 1 開始）

現有 skill / tool 存在下列缺口，若不先處理，Stage 1 起跑即踩雷；此外，本 repo 本身也需要一套可被 agent 自動遵循的**規劃開發骨架**：

- **命名錯位**：`picoclaw-ops-companion` 實質是 2FA copilot CLI gating；命名暗示它屬於 picoclaw 生態；建議 refine + rename 為 `ops-companion`（與 Stage 6 安全治理對齊）
- **codex-* 命名**：`codex-lesson`、`codex-project-insights` 雖可跨 provider 使用，但名稱綁死 codex；需 rename（例如 `session-lesson`、`project-insights`）並補 Claude Code session file 支援
- **無 Claude Code 支援**：`session-health`、`coordinator`、`codex-lesson/insights` 的 session 解析對 Claude Code 的 session 檔格式支援不足
- **AGENTS.md 擴散**：目前 AGENTS.md / INSTRUCTION.md / CLAUDE.md 各自為政，需定 canonical + include 規則
- **Stage 2 atomize 邊界未定稿**：obs-auto-moc 已提供 `atomized_from` provenance + Decayed + `record-agent-reference` reactivation，但原子化**執行**在 PicoClaw (pi3 live runtime)，非 obs-auto-moc 本體；兩邊 Stage 1/2/3 編號不同語意，需本 repo 統一 convention
- **規劃開發流程未骨架化**：本 repo 打算採 **openspec + superpowers** 做 change management + 創作流程；需先把目錄、AGENTS.md、`opsx-*` prompt 等鋪設完成，後續 stage 才能用 `brainstorming → writing-plans → openspec propose → subagent-driven-development → archive` 這條流水線

**Stage 0 輸出**：

1. rename / refine 清單（PR 到各 skill repo）
2. Claude Code session 格式 adapter
3. 各 skill 的 version pin 表，寫進 `paulshaclaw/config/skills.lock.yaml`
4. 一份 tool matrix（改寫為 `openspec/specs/stage0/tool-matrix.md`）
5. Stage 2 / PicoClaw 邊界聲明：paulshaclaw 的 Stage 2 spec 裡寫死「`obs-auto-moc`（rename 目標：`paulsha-memory`）維 provenance + Decayed；PicoClaw on pi3 執行 atomize；`ops-companion` 不是 Stage 2 runtime」；並於 source repo 加 convention 前綴（`obs-auto-moc.StageN` vs `paulshaclaw.StageN`）避開編號衝突
6. **openspec + superpowers 骨架**：在本 repo 根目錄建立 `openspec/{specs,changes}/`、`docs/superpowers/{specs,plans}/`、根目錄 `AGENTS.md`，作為後續所有 stage 的 change / design / plan 入口樣版
7. `/opsx:new`、`/opsx:ff` 文件與 prompt 必須要求寫入邊界、跨 stage 依賴、測試 gate、證據路徑，且 `using-git-worktrees` 流程需檢查 branch/worktree 對照是否可追到 `origin/<branch>`

### 5.3 各 stage 定位（一句話）

- **Stage 0**：把地基的工具擦亮、對齊命名、補 Claude Code 支援
- **Stage 1**：把人類操作介面（TUI + Telegram）與控制核心（PaulShiaBro daemon + tmux-bridge + coordinator）做起來。**`/dispatch <task_id>` 是手動派工路徑;coordinator（多 agent 拆 todo→分派→整合）+ §3.1 hub-and-spoke `multi_agent_devflow` 政策由 Stage 1 擁有,規劃 vendored 進 paulshaclaw(ref skill `custom-skills/coordinator` → paulshaclaw flow),不長期漂浮在外部 skill + CLAUDE.md**
- **Stage 2**：把 agent 的記憶從「散在 session / 筆記」收斂成分層可治理的結構
- **Stage 3**：把「自然語言下指令」轉成「artifact-driven lifecycle」
- **Stage 4**：把「誰有資格做什麼」正式化成 persona contract
- **Stage 5**：看得見、修得了、追得到錯誤
- **Stage 6**：該擋的動作擋住、該留的痕跡留下
- **Stage 7**：能從零台機器裝起來

## 5.5 實證 devflow（manager × builder × reviewer，consume Stage 3 phases）

近輪（skillopt / lifecycle-fix / T6）驗證、之後 Stage 3/4 應固化的具體流程:

| 角色（Stage 4 persona） | phase（Stage 3） | 實際執行 |
|---|---|---|
| **manager**（人類在環） | research / define / plan | `brainstorming → design spec → openspec-propose → writing-plans`,於 `feature/<slug>` 分支(R-12,無點) |
| **builder**（委派） | build | `copilot --model gpt-5.4 --yolo -p {PROMPT}`:在分支依 plan 跑 TDD/subagent-driven → requesting-code-review → openspec-archive → policy/commit/push/PR |
| **reviewer**（人類在環,review gate） | verify / review / ship | 深度 code-review(找/修真 bug)→ 全測試 + CI policy 綠 → SHA 比對確認 review==merge → squash merge + delete-branch → `pull --ff-only` + prune |

跨切面紀律(寫進各 stage 契約):determinism（注入 `now`)、fail-open vs fail-closed 依 artifact 關鍵度、零重複/重用、vendored 程式逐字一致(改進先進 upstream)、systematic-debugging(先根因後修)、SHA-verify-before-merge。**reviewer 是否可改 code 仍待 Stage 4 拍板(A 擴權 / B 唯讀退回 builder / C 有界修正)。**

## 5.6 Workflow 模式庫與選擇器（架構提升，planned）

來源：6 種 agent workflow 模式整理（`obs_vault/000-paulshaclaw-task-mngr-use.md`）。核心論點：**workflow 不是 agent 越多越好，而是依任務型態選對「分工 / 驗證 / 收斂」方式**。

**現況落差**：目前只跑**單一寫死的 devflow**（§5.5：manager → 單 builder → 單 reviewer，順序單線）。`§3.1 hub-and-spoke` + `multi_agent_devflow` 已預留多模式，**尚未實現**。把這 6 種當成一個 **pattern 庫**，由 manager/coordinator **依任務型態選**，是把架構從「一種固定流程」升級成「多 workflow + 選擇器」的根本提升。

| # | 模式 | 對 paulshaclaw 的意義 | 落點 | 狀態 |
|---|---|---|---|---|
| 1 | Classify-And-Act | meta 應用：依任務型態**選 workflow**（不是永遠單線） | Stage 3 lifecycle + coordinator | [ ] (已規劃，**最高槓桿**) |
| 2 | Fanout-And-Synthesize | 可分解 stage 用**並行 builder（worktree）→ 整合**，實現 `multi_agent_devflow` | coordinator | [ ] (已規劃) |
| 3 | Adversarial Verification | review gate 引入**多獨立 verifier**（skillopt/lifecycle/T9 三個 critical bug 為單一 reviewer 風險之證） | Stage 4 reviewer | [ ] (已規劃) |
| 4 | Generate-And-Filter | 設計階段 rubric 篩選；SkillOpt 已是 generate→gate | Stage 3/4 | [~] (部分：SkillOpt) |
| 5 | Tournament | 利基（設計稿兩兩比稿） | — | [ ] (YAGNI，暫不採) |
| 6 | Loop-Until-Done | systematic-debugging / SkillOpt loop 已具備 | — | [x] (大致涵蓋) |

**最高槓桿三項**：① workflow 選擇器（模式 1 meta）、② adversarial review gate（模式 3）、③ fanout 實現 `multi_agent_devflow`（模式 2）。模式 5（Tournament）/ 4（Generate-And-Filter 全量）先不採（YAGNI），模式 6 已大致涵蓋。實作落點與 §5.3（coordinator vendoring）、Stage 3（lifecycle）、Stage 4（reviewer persona）對齊。

## 7. Stage 獨立性 contract

### 7.1 規則

**後面 stage 失敗，前面 stage 必須能繼續運作**（修復優先，不 cascade）。

### 7.2 每個 stage 的「下游失敗」降級表

| 本 stage | 下游失敗 | 降級行為 |
|---|---|---|
| Stage 1 | Stage 2 memory 掛 | 繼續運行；memory_loadout 只載 common-sense 預設；inbox 改寫本地 `runtime/fallback-inbox/`，待 Stage 2 恢復後 replay |
| Stage 1 | Stage 3 lifecycle 掛 | 繼續運行；TUI 可手動呼叫 coordinator job（不經 slash gate） |
| Stage 1 | Stage 4 persona 掛 | 繼續運行；worker 啟動帶 `default-unconstrained` persona，log 警告 |
| Stage 1 | Stage 5 觀測掛 | 繼續運行；log 本地 fallback，Operator 只能靠 CLI `psc status` 查狀態 |
| Stage 1 | Stage 6 security 掛 | **fail-close**：任何高風險 action（push、deploy）停擺；低風險照常 |
| Stage 2 | Stage 5 觀測掛 | dream service 照跑；health metric 本地落盤，待重啟時重算 |
| Stage 3 | Stage 4 persona 掛 | gate 只做 static schema 檢查，persona check 跳過並標 `guardrail: skipped` |
| Stage 3 | Stage 5 觀測掛 | events.jsonl 本地寫入照舊；dashboard 一時看不到，不影響 gate |
| Stage 4 | Stage 5 觀測掛 | guardrail 照常；drift signal 延遲 |

### 7.3 三個 always-on 元件必須**失敗域分離**

1. PaulShiaBro daemon（含 Manager orchestrator）
2. Telegram bot listener（Stage 1 infra，Operator 的遠端入口）
3. Memory Janitor service（Stage 2 背景整理）

**不共享 process、不共享 lock、不共享 watchdog**。任何一個 crash，另外兩個照常運作。

### 7.4 Tmux server 是例外

tmux server 死掉 = 整個 paulshaclaw 重啟。不做 tmux-less fallback。Stage 5 supervisor 偵測 tmux 死亡時觸發 full restart sequence。

- [ ] User overlay（從 obs-auto-moc `distill-persona-goals`）存在且有 checksum
- [ ] Stage 3 gate 多一線：提交者 persona 在 `allowed_phases` 中
- [ ] Persona A/B shadow run 可選擇開（非必要）
- [ ] **Slice roster 與 co-worker awareness（見 `04` §9.5.6）**：`runtime/lifecycle/slices/<slice_id>/roster.yaml` 由 coordinator 單點寫入；agent bind 時注入 `<co_workers>` block、task envelope 帶 `context.peers`、跨 slice 任務拒收、同 slice claim 互斥、幽靈 handoff 改派或升級、長跑 agent re-sync、coordinator 掛時降級只讀；golden slice 回歸測試覆蓋 R1–R12

### Stage 5 — 觀測 / 健康 / failover / log

**目標**：**看得見、修得了、追得到錯誤**；daemon crash、tmux 死、bot 斷線都有可靠復原手段。

**驗收標準**：

- [ ] Health probe：三個 always-on 元件各自回報 `alive / last_tick / queue_depth`
- [ ] Supervisor：daemon / bot listener / Memory Janitor 任一 crash 由 supervisor 重啟，重啟事件寫 `runtime/health/incidents.jsonl`
- [ ] **Failover 策略**：daemon crash 時 registry 從 `~/.agents/state/registry-snapshot.jsonl` 重建；最近 N job 可 replay
- [ ] Tmux server 死 → supervisor 觸發 full restart（tmux → daemon → workers），Operator 收通知
- [ ] **Log 周期 = lifecycle slice**：slice 結束時，成功 → 摘要入 `experience/<slice>.md` 後丟 raw；失敗 → raw log 保留於 `experience/incidents/<slice>/`
- [ ] 錯誤 log 追蹤：`runtime/errors/*.jsonl` 含 actor / stage / phase / hint；Stage 6 audit trail 可索引此處
- [ ] Dashboard：federated（TUI + session-health HTML + obs overlays）；**不做新 web UI**
- [ ] Recovery playbook：`docs/ops/recovery.md` 列出每種 crash 的手動復原步驟

### Stage 6 — 安全與機敏治理（ops-companion 核心）

**目標**：高風險動作必經 approval；secret 不外流；審計可回追。

**驗收標準**：

- [ ] `picoclaw-ops-companion` 已於 Stage 0 rename 為 `ops-companion`
- [ ] Approval gate：`git push / deploy / package install / remote op` 全部走此 gate；無 approval 拒絕
- [ ] 2FA 機制：approval 要求 Operator 二次確認（Telegram 按鈕 / CLI prompt）
- [ ] Secret redaction：inbox importer 在寫 `work-centric/` 前過 redactor；命中規則（token、cookie、private URL、email）→ 替換 placeholder + 寫 `runtime/redacted/*`
- [ ] Classification：每份 artifact frontmatter 帶 `classification: public | internal | confidential | restricted`；classifier 依規則預填、Operator 可覆寫
- [ ] External publish guard：對外送檔前檢查 classification；`confidential / restricted` 禁發
- [ ] Audit trail：所有 approval / override / secret 命中寫 `~/.agents/runtime/audit/*.jsonl`，不可刪除（append-only）
- [ ] `~/.config/paulshaclaw/` 權限 `0700`；啟動時檢查、權限不符拒絕啟動

### Stage 7 — 部署（core / state / secret）

**目標**：一條命令從乾淨機器裝起全套；升級不蓋使用者資料。

**驗收標準**：

- [ ] `psc install` 指令存在；可分 `--core-only / --state-only / --secret-only / --all`
- [ ] Core 安裝：把 `paulshaclaw/` 放到 `/opt/paulshaclaw/` 或 `~/.local/share/paulshaclaw/`；systemd unit 安裝到 user scope
- [ ] State 安裝：若 `~/.agents/` 不存在 → 從 `paulshaclaw/templates/agents/` rename 部署；若存在 → skip 並記 log
- [ ] Memory symlink 安裝：檢查 `~/notes/paulshaclaw/memory` 存在；不存在則提示建立、不強制；存在則自動建 symlink
- [ ] Secret 安裝：`psc install --secret-only` 走互動式流程（botfather 半自動 / user id / token paste）；寫到 `~/.config/paulshaclaw/*.env`、權限 `0600`
- [ ] `.sample` / `.template` 後綴 convention 統一；rename 規則：刪掉後綴即目標檔名
- [ ] 升級（`psc upgrade`）：只覆寫 core，不動 state / secret；diff 報告列給 Operator 確認
- [ ] Uninstall：`psc uninstall` 只移 core 與 systemd unit；state / secret 保留（提示 Operator 手動處理）

## 9. 觀測 / 失敗 / log 總原則

（濃縮 Stage 5 的跨層規則，便於其他 stage 設計時引用）

1. **每個 always-on service 必須 expose liveness + ready endpoint**（local unix socket 即可）
2. **log 依 lifecycle slice 為生命週期**；成功即摘要＋丟原始，失敗保留
3. **Supervisor 是唯一可重啟 service 的身份**；persona / skill / agent instance 沒有自殺 / 自啟權限
4. **錯誤歸因前先記錄**：一個異常先寫 `runtime/errors/*.jsonl`，再決定是否重試 / 升級
5. **不自動重試**；重試要由 Operator 或明確 policy 觸發
6. **Tmux 死 = 整體重啟**；不做無縫接手

## 10. 符號 / 版本 / 目錄 convention

### 10.1 版本

- 專案整體 semver：major 改動（stage 引進 / 捨棄）、minor（stage 內功能）、patch（修文件 / 修 bug）
- 每個 stage 自己也 semver（`stage1/0.2.0`）；stage 版本獨立 tag：`stage1-v0.2.0`
- `personas.yaml` / `lifecycle.yaml` schema 獨立 semver

### 10.2 目錄

- 程式 / template → `paulshaclaw/`
- state / memory → `~/.agents/`
- secret → `~/.config/paulshaclaw/`
- runtime（跑起來才有的）→ 各 service `runtime/` 子目錄，`.gitignore`
- canonical spec + change management → `openspec/{specs,changes}/`（Stage 0 建立）
- 自由探索 → `docs/research/`
- 創作設計 + 實作計劃 → `docs/superpowers/{specs,plans}/`（Stage 0 建立）
- 運維手冊 → `docs/ops/`
- agent workflow 入口 → 根目錄 `AGENTS.md`（`CLAUDE.md` 為其 alias）

### 10.3 命名

- CLI 子命令：`psc <verb>`，動詞一字（`psc status`、`psc install`、`psc ship`）
- systemd unit：`paulshiabrod.service`、`paulshiabro-bot.service`、`paulshiabro-janitor.service`
- log prefix：`[psc.<component>]`
- events kind：`snake_case`（`lifecycle.phase.requested`、`guardrail.denied`）
